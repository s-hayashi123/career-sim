## Learn（概念理解）

### 4回目のレトロで意識すること

Project 01, 02, 03 との違い:
- **P01**: 環境構築・基本ワークフロー
- **P02**: バグ修正・再発防止
- **P03**: 機能開発（ゼロから設計・実装）
- **P04**: リファクタリング・保守性改善 ← 今回

### リファクタリングの振り返り

**「なぜリファクタリングしたか」を明確に**:

```markdown
## Good
❌ 「コードを整理した」
✅ 「サービス層を導入し、ルートハンドラが 50行から 5-10行に削減。
   ビジネスロジックが HTTP から独立し、単体テストが可能になった」

❌ 「型ヒントを追加した」
✅ 「mypy を導入し、実行前に型エラーを検出できるようになった。
   IDE の補完も効くようになり、開発効率が向上」
```

### リファクタリングの成果の定量化

Before / After で比較:

| 指標 | Before (P03終了時) | After (P04終了時) | 改善 |
|------|------------------|------------------|------|
| main.py の行数 | 1000行 | 200行 | -80% |
| ルートハンドラの平均行数 | 30行 | 7行 | -77% |
| 重複コード（バリデーション） | 5箇所 | 0箇所 | -100% |
| mypy エラー | - | 0 | ✅ 導入 |
| サービス層の単体テスト | 0 | 50+ | ✅ 追加 |

### 100本ノック全体の振り返り

P01-P04 を完走した時点で:

- **P01**: 基本スキル習得（環境構築、Git、PR）
- **P02**: 問題解決スキル（バグ修正、テスト、ADR）
- **P03**: 機能開発スキル（設計、実装、E2E）
- **P04**: 保守性改善スキル（リファクタリング、DI、型）

**次のステップ**:
- P05 以降（もしあれば）
- 実際のプロジェクトへの応用

## Practice（練習課題）

### 課題: 以下の振り返りを改善せよ

```markdown
## P03 の Try の振り返り
（空欄）

## Good
- サービス層を作った
- 型ヒントを追加した
- ドキュメントを更新した

## Bad
- 時間がかかった

## Try
- 引き続き頑張る
```

**問題点を指摘し、改善せよ。**

### 模範解答

<details>
<parameter name="summary">クリックで表示</summary>

**問題点**:
1. P03 の Try を振り返っていない
2. Good が抽象的（何をどう作った/追加したか不明）
3. Bad が「時間がかかった」だけ（なぜ、どこで？）
4. Try が「頑張る」で具体的なアクションがない

**改善**:

```markdown
## P03 の Try の振り返り
- 「E2E テストを先に設計してから実装する」→ ✅ P04-02 で実践できた
- 「ADR を書くタイミングを早める」→ ❌ ADR-006 の作成が遅れた

## Good
- サービス層導入で main.py が 1000行から 200行に削減（-80%）
- ルートハンドラが 5-10行に統一され、見通しが良くなった
- mypy 導入で IDE の補完が効くようになり、型エラーを事前検出
- 統一レスポンスモデルでエラーハンドリングが一貫した
- BaseModel ミックスインで重複フィールド（id, created_at）を削減

## Bad
- サービス層の抽出で 6時間かかった（想定 4時間）
  → 既存のテストが壊れる箇所が多く、修正に時間を取られた
- mypy のエラー修正に苦戦（特に Optional の扱い）
  → 公式ドキュメントを読むべきだった
- ADR-006 の作成が後回しになり、設計判断の記録が遅れた

## Try
- リファクタリング前に既存テストを全て通すことを確認する
- mypy のエラーは出た時点で公式ドキュメントを 10分読む
- ADR はコードを書く前に書く（設計判断を先に記録）
- 定量的な改善指標（行数削減率など）を意識する
```

</details>
