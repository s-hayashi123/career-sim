# P04-01: サービス層分離方針（ADR-006）

> **シナリオ**: TLから「ルートハンドラが肥大化。サービス層を分離したい」
> **想定時間**: 4h
> **スキルタグ**: P6（リファクタリング）, P8（ドキュメント・プロセス改善）

---

## Learn（概念理解）

### なぜサービス層が必要か

コードベースが成長すると、**ルートハンドラ（コントローラ）が肥大化**する:

```python
# ❌ Bad: ファットコントローラ（全ロジックがルートハンドラに）
@app.post("/users")
def create_user(user: UserCreate, db: Session = Depends(get_db)):
    # バリデーション
    if len(user.name) < 2:
        raise HTTPException(status_code=422, detail="Name too short")

    # 重複チェック
    existing = db.query(User).filter(User.email == user.email).first()
    if existing:
        raise HTTPException(status_code=409, detail="Email already exists")

    # DB操作
    db_user = User(name=user.name, email=user.email)
    db.add(db_user)

    try:
        db.commit()
    except IntegrityError:
        db.rollback()
        raise HTTPException(status_code=409, detail="Email conflict")

    db.refresh(db_user)

    # 通知送信
    send_welcome_email(user.email)

    # ログ記録
    log_user_creation(db_user.id)

    return db_user
```

**問題点**:
- ロジックが散在してテストしにくい
- 同じロジック（重複チェックなど）を他のエンドポイントでもコピペ
- ビジネスロジックと HTTP レイヤーが密結合
- コントローラが50行、100行と肥大化

### 3層アーキテクチャ

**責任を分離する**設計パターン:

```
┌─────────────────┐
│  Presentation   │  ← ルートハンドラ（薄く保つ）
│   (Routes)      │     リクエスト受け取り、レスポンス返す
└────────┬────────┘
         │
┌────────▼────────┐
│   Business      │  ← サービス層（ビジネスロジック）
│   (Services)    │     バリデーション、DB操作、外部API呼び出し
└────────┬────────┘
         │
┌────────▼────────┐
│   Data Access   │  ← モデル層（DB定義）
│   (Models)      │     SQLAlchemy モデル
└─────────────────┘
```

### サービス層の実装パターン

```python
# ✅ Good: サービス層にロジックを分離

# services/user_service.py
class UserService:
    def __init__(self, db: Session):
        self.db = db

    def create_user(self, user_data: UserCreate) -> User:
        """ユーザー作成のビジネスロジック"""
        # 重複チェック
        if self._email_exists(user_data.email):
            raise ValueError("Email already exists")

        # DB操作
        db_user = User(name=user_data.name, email=user_data.email)
        self.db.add(db_user)
        self.db.commit()
        self.db.refresh(db_user)

        # 副作用（通知、ログなど）
        self._send_welcome_email(db_user.email)
        self._log_creation(db_user.id)

        return db_user

    def _email_exists(self, email: str) -> bool:
        return self.db.query(User).filter(User.email == email).first() is not None

    def _send_welcome_email(self, email: str):
        # メール送信ロジック
        pass

    def _log_creation(self, user_id: int):
        # ログ記録
        pass

# main.py (ルートハンドラは薄く)
@app.post("/users", status_code=201)
def create_user(user: UserCreate, db: Session = Depends(get_db)):
    service = UserService(db)
    try:
        db_user = service.create_user(user)
        return db_user
    except ValueError as e:
        raise HTTPException(status_code=409, detail=str(e))
```

**メリット**:
- ルートハンドラが薄い（5-10行）
- ビジネスロジックがテストしやすい（HTTP から独立）
- ロジックの再利用が容易
- 責任が明確

### リファクタリングの手順

```
1. 現状分析 → どのロジックがファット化しているか特定
2. ADR 作成 → サービス層の方針を記録
3. サービス層実装 → 既存のロジックを移動
4. テスト → 全テストが通ることを確認
5. 段階的に適用 → 全エンドポイントを一度に変えない
```

### ADR での設計判断

```markdown
# ADR-006: サービス層の導入

## Context
- ルートハンドラが肥大化（50行超）
- 同じロジックが複数エンドポイントにコピペ
- テストが HTTP レイヤーに依存

## Decision
3層アーキテクチャを採用:
- Routes: リクエスト/レスポンス処理のみ（薄く）
- Services: ビジネスロジック
- Models: DB 定義

## Consequences
**Pros**:
- テスタビリティ向上
- コードの再利用性向上
- 責任が明確

**Cons**:
- レイヤーが増えて複雑化
- 小規模なロジックにはオーバーヘッド
```

## Practice（練習課題）

### 課題: 以下のファットコントローラをリファクタリングせよ

```python
@app.post("/orders")
def create_order(order: OrderCreate, db: Session = Depends(get_db)):
    # 在庫チェック
    product = db.query(Product).filter(Product.id == order.product_id).first()
    if not product:
        raise HTTPException(status_code=404, detail="Product not found")
    if product.stock < order.quantity:
        raise HTTPException(status_code=422, detail="Insufficient stock")

    # 注文作成
    db_order = Order(
        product_id=order.product_id,
        quantity=order.quantity,
        total=product.price * order.quantity
    )
    db.add(db_order)

    # 在庫減少
    product.stock -= order.quantity

    db.commit()
    db.refresh(db_order)

    # 通知
    send_order_confirmation(db_order.id)

    return db_order
```

**Q1.** このコードの問題点を3つ挙げよ。
**Q2.** OrderService にどのメソッドを作るべきか？
**Q3.** リファクタリング後のルートハンドラを書け。

### 模範解答

<details>
<summary>クリックで表示</summary>

**A1. 問題点**:
1. ビジネスロジック（在庫チェック、在庫減少）がルートハンドラに直接書かれている
2. 複数の DB 操作が1つの関数にまとまっていてテストしにくい
3. 通知送信などの副作用もルートハンドラで実行している

**A2. OrderService のメソッド**:
- `create_order(order_data: OrderCreate) -> Order`（メイン処理）
- `_check_product_availability(product_id, quantity) -> Product`（在庫確認）
- `_reduce_stock(product: Product, quantity: int)`（在庫減少）
- `_send_confirmation(order_id: int)`（通知送信）

**A3. リファクタリング後**:

```python
# services/order_service.py
class OrderService:
    def __init__(self, db: Session):
        self.db = db

    def create_order(self, order_data: OrderCreate) -> Order:
        # 在庫チェック
        product = self._check_product_availability(
            order_data.product_id,
            order_data.quantity
        )

        # 注文作成
        db_order = Order(
            product_id=order_data.product_id,
            quantity=order_data.quantity,
            total=product.price * order_data.quantity
        )
        self.db.add(db_order)

        # 在庫減少
        self._reduce_stock(product, order_data.quantity)

        self.db.commit()
        self.db.refresh(db_order)

        # 通知
        self._send_confirmation(db_order.id)

        return db_order

    def _check_product_availability(self, product_id: int, quantity: int) -> Product:
        product = self.db.query(Product).filter(Product.id == product_id).first()
        if not product:
            raise ValueError("Product not found")
        if product.stock < quantity:
            raise ValueError("Insufficient stock")
        return product

    def _reduce_stock(self, product: Product, quantity: int):
        product.stock -= quantity

    def _send_confirmation(self, order_id: int):
        send_order_confirmation(order_id)

# main.py
@app.post("/orders", status_code=201)
def create_order(order: OrderCreate, db: Session = Depends(get_db)):
    service = OrderService(db)
    try:
        db_order = service.create_order(order)
        return db_order
    except ValueError as e:
        # ビジネスロジックのエラーを HTTP エラーに変換
        if "not found" in str(e):
            raise HTTPException(status_code=404, detail=str(e))
        else:
            raise HTTPException(status_code=422, detail=str(e))
```

</details>

## Mission（本番課題）

### 状況

TL から以下のメッセージが来ました:

> 「P03 で機能追加してきたけど、main.py が 1000行超えてきて見通しが悪い。ルートハンドラにロジックが詰まりすぎてる。サービス層を導入したいので、まず方針を ADR-006 としてまとめてください。
>
> 現状の問題点を分析して:
> - どのエンドポイントがファット化しているか
> - どんなロジックが重複しているか
> - サービス層をどう分けるか（UserService, ProjectService など）
> - 段階的な移行計画」

### やること

1. **現状のコードを分析する**
   - main.py を読んで、各ルートハンドラの行数を確認
   - ファット化しているエンドポイントを特定（30行超など）
   - 重複しているロジックを洗い出す

2. **GitHub Issue を作成する**
   - タイトル: `refactor: サービス層を導入する`
   - 本文: 現状の問題点、導入する方針

3. **ブランチを作成する**
   ```bash
   git checkout -b refactor/proj04-service-layer-adr
   ```

4. **ADR-006 を作成する**
   - ファイル: `docs/adrs/ADR-006-service-layer-introduction.md`
   - Status: Accepted
   - Context: 現状の問題点（ファットコントローラ、重複ロジック）
   - Decision: 3層アーキテクチャ採用、サービス層の責任範囲
   - Consequences: メリット・デメリット

5. **ADR に以下を含める**
   - サービス層の設計方針
     - `services/user_service.py`
     - `services/project_service.py`
     - `services/task_service.py`
   - 各サービスの責任範囲
   - ルートハンドラのあるべき姿（5-10行）
   - 移行計画（どの順番でリファクタリングするか）

6. **ディレクトリ構造を提案する**
   ```
   project-atlas-api/
   ├── main.py              # ルートハンドラのみ（薄く）
   ├── models.py            # SQLAlchemy モデル
   ├── schemas.py           # Pydantic スキーマ
   ├── services/
   │   ├── __init__.py
   │   ├── user_service.py
   │   ├── project_service.py
   │   └── task_service.py
   └── tests/
   ```

7. **PR を作成する**

<summary>ADR-006 のテンプレート</summary>

```markdown
# ADR-006: サービス層の導入

## Status
Accepted

## Context

### 現状の問題点
- main.py が 1000行超で見通しが悪い
- ルートハンドラが肥大化（30-50行）
- 同じロジックが複数箇所にコピペされている:
  - 外部キー存在確認（project_id, assignee_id）
  - IntegrityError ハンドリング
  - ステータス遷移バリデーション
- テストが HTTP レイヤーに依存（ビジネスロジックの単体テストが困難）

### ファット化しているエンドポイント
- POST /tasks: 40行（外部キー検証、作成、履歴記録）
- PUT /tasks/{id}: 35行（ステータス遷移、更新、履歴記録）
- GET /projects/{id}/summary: 30行（複雑な集計ロジック）

## Decision

### 3層アーキテクチャの採用
```
Routes (main.py)     → リクエスト/レスポンス処理のみ
Services (services/) → ビジネスロジック
Models (models.py)   → DB 定義
```

### サービス層の責任
- ビジネスロジックの実行
- DB 操作
- バリデーション
- 外部サービス呼び出し（メール送信など）

### ルートハンドラの責任
- リクエストの受け取り
- サービスの呼び出し
- レスポンスの返却
- HTTP エラーへの変換

### サービスの分割方針
- `UserService`: ユーザー関連のロジック
- `ProjectService`: プロジェクト関連のロジック
- `TaskService`: タスク関連のロジック

### 移行計画
1. P04-02: UserService 実装
2. P04-03: ProjectService, TaskService 実装
3. 各ステップでテストが通ることを確認

## Consequences

**Pros**:
- ルートハンドラが薄くなり、見通しが良くなる
- ビジネスロジックが HTTP から独立し、テストしやすくなる
- ロジックの再利用が容易
- 責任が明確になる

**Cons**:
- レイヤーが増えて、ファイル数が増加
- 小規模なロジックにはオーバーヘッド
- 学習コストが上がる

**リスク**:
- 移行中にバグを混入させる可能性 → 各ステップで全テストを実行
```

## Review（自己採点チェックリスト）

### 必須項目

- [ ] ADR-006 が作成されている
- [ ] 現状の問題点が具体的に記載されている（ファット化、重複ロジック）
- [ ] 3層アーキテクチャの採用が明記されている
- [ ] サービス層とルートハンドラの責任範囲が定義されている
- [ ] サービスの分割方針（UserService, ProjectService, TaskService）がある
- [ ] 移行計画が含まれている
- [ ] Consequences（Pros/Cons）が記載されている
- [ ] GitHub Issue と PR が作成されている

### 加点項目

- [ ] 現状のコード分析（行数、重複箇所）が定量的
- [ ] ディレクトリ構造の提案がある
- [ ] サービス層のテスト方針が記載されている
- [ ] リスクと対策が明記されている

### よくある間違い

- 現状分析をせずに、一般論だけで ADR を書く
- サービス層の責任範囲が曖昧（どこまでサービスに入れるか不明確）
- 移行計画がない（一度に全て変えようとする）
- Consequences が抽象的（「保守性が上がる」だけ）
- テスト方針を考慮していない
- ルートハンドラが「薄く」あるべき理由を説明していない
