## Learn（概念理解）

### API レスポンスの標準化

現状の問題:

```python
# ❌ Bad: エラー形式がバラバラ
# エンドポイント A
raise HTTPException(status_code=404, detail="User not found")
# → {"detail": "User not found"}

# エンドポイント B
raise HTTPException(status_code=422, detail={"field": "email", "message": "Invalid"})
# → {"detail": {"field": "email", "message": "Invalid"}}

# エンドポイント C
return {"error": "Something went wrong"}
# → {"error": "Something went wrong"}

# クライアントが混乱: detail? error? 文字列? オブジェクト?
```

**統一レスポンス**の設計:

```python
# ✅ Good: 統一形式
{
  "success": false,
  "error": {
    "code": "NOT_FOUND",
    "message": "User not found",
    "details": {"resource": "User", "id": 123}
  }
}

{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Invalid input",
    "details": {
      "field": "email",
      "reason": "Invalid email format"
    }
  }
}
```

### 統一レスポンスモデルの実装

```python
# schemas.py
from pydantic import BaseModel
from typing import Any

class ErrorDetail(BaseModel):
    """エラー詳細"""
    code: str  # エラーコード（NOT_FOUND, VALIDATION_ERROR など）
    message: str  # 人間が読めるメッセージ
    details: dict[str, Any] | None = None  # 追加情報

class ErrorResponse(BaseModel):
    """エラーレスポンス"""
    success: bool = False
    error: ErrorDetail

class SuccessResponse(BaseModel):
    """成功レスポンス（汎用）"""
    success: bool = True
    data: Any

# 使用例
@app.get("/users/{user_id}")
def get_user(user_id: int, db: Session = Depends(get_db)):
    service = UserService(db)
    try:
        user = service.get_user_by_id(user_id)
        return SuccessResponse(data=user)
    except ValueError:
        return ErrorResponse(
            error=ErrorDetail(
                code="NOT_FOUND",
                message=f"User with id {user_id} not found",
                details={"resource": "User", "id": user_id}
            )
        )
```

### グローバル例外ハンドラ

FastAPI で**統一エラーハンドリング**:

```python
from fastapi import Request
from fastapi.responses import JSONResponse

# カスタム例外
class ResourceNotFoundError(Exception):
    def __init__(self, resource: str, id: int):
        self.resource = resource
        self.id = id

class ValidationError(Exception):
    def __init__(self, field: str, message: str):
        self.field = field
        self.message = message

# グローバル例外ハンドラ
@app.exception_handler(ResourceNotFoundError)
async def resource_not_found_handler(request: Request, exc: ResourceNotFoundError):
    return JSONResponse(
        status_code=404,
        content={
            "success": False,
            "error": {
                "code": "NOT_FOUND",
                "message": f"{exc.resource} not found",
                "details": {"resource": exc.resource, "id": exc.id}
            }
        }
    )

@app.exception_handler(ValidationError)
async def validation_error_handler(request: Request, exc: ValidationError):
    return JSONResponse(
        status_code=422,
        content={
            "success": False,
            "error": {
                "code": "VALIDATION_ERROR",
                "message": exc.message,
                "details": {"field": exc.field}
            }
        }
    )

# サービス層で使用
class UserService:
    def get_user_by_id(self, user_id: int) -> User:
        user = self.db.query(User).filter(User.id == user_id).first()
        if not user:
            raise ResourceNotFoundError("User", user_id)  # カスタム例外
        return user
```

### エラーコードの体系化

```python
# constants/error_codes.py
class ErrorCode:
    # 4xx Client Errors
    NOT_FOUND = "NOT_FOUND"
    VALIDATION_ERROR = "VALIDATION_ERROR"
    CONFLICT = "CONFLICT"
    UNAUTHORIZED = "UNAUTHORIZED"
    FORBIDDEN = "FORBIDDEN"

    # 5xx Server Errors
    INTERNAL_ERROR = "INTERNAL_ERROR"
    DATABASE_ERROR = "DATABASE_ERROR"

# 使用
ErrorDetail(
    code=ErrorCode.NOT_FOUND,
    message="User not found"
)
```

## Practice（練習課題）

### 課題: 以下のエラーレスポンスを統一形式に変換せよ

現状:

```python
# エンドポイント 1
raise HTTPException(status_code=404, detail="Product not found")

# エンドポイント 2
return {"error": "Stock insufficient", "available": 5}

# エンドポイント 3
raise HTTPException(status_code=422, detail={"field": "price", "error": "Must be positive"})
```

**Q1.** ErrorResponse モデルを定義せよ。
**Q2.** 各エラーを統一形式に変換せよ。
**Q3.** グローバル例外ハンドラを書け。

## Mission（本番課題）

### 状況

TL から以下のフィードバックが来ました:

> 「エラーレスポンスの形式が統一されていない。クライアント側でエラーハンドリングしにくいと言われた。統一レスポンスモデルを導入して、ADR-007 にまとめてください。」

### やること

1. **GitHub Issue を作成する**
   - タイトル: `refactor: 統一レスポンスモデルを導入`
   - 本文: 現状の問題点、統一形式の提案

2. **ブランチを作成する**
   ```bash
   git checkout -b refactor/proj04-unified-response
   ```

3. **統一レスポンスモデルを定義する**
   - `schemas.py` に追加:
     - `ErrorDetail`: エラー詳細（code, message, details）
     - `ErrorResponse`: エラーレスポンス（success=False, error）
     - `SuccessResponse`: 成功レスポンス（success=True, data）

4. **エラーコードを定義する**
   - ファイル: `constants/error_codes.py`
   - クラス: `ErrorCode`
     - NOT_FOUND, VALIDATION_ERROR, CONFLICT, UNAUTHORIZED など

5. **カスタム例外を作成する**
   - ファイル: `exceptions.py`
   - 例外クラス:
     - `ResourceNotFoundError(resource, id)`
     - `ValidationError(field, message)`
     - `ConflictError(message)`

6. **グローバル例外ハンドラを追加する**
   - `main.py` に追加
   - 各カスタム例外に対応するハンドラ

7. **サービス層をリファクタリングする**
   - ValueError → カスタム例外に変更
   - 統一されたエラーハンドリング

8. **ADR-007 を作成する**
   - ファイル: `docs/adrs/ADR-007-unified-response-model.md`
   - Context: 現状の問題点
   - Decision: 統一レスポンス形式、エラーコード体系
   - Consequences: メリット・デメリット

9. **既存のテストを更新する**
   - エラーレスポンスの形式が変わるため、テストを修正

10. **全テストが通ることを確認する**
    ```bash
    pytest tests/ -v
    ```

11. **PR を作成する**

# exceptions.py
class ResourceNotFoundError(Exception):
    def __init__(self, resource: str, resource_id: int):
        self.resource = resource
        self.resource_id = resource_id
        super().__init__(f"{resource} with id {resource_id} not found")

class ValidationError(Exception):
    def __init__(self, field: str, message: str):
        self.field = field
        self.message = message
        super().__init__(f"Validation error on {field}: {message}")

class ConflictError(Exception):
    def __init__(self, message: str, details: dict | None = None):
        self.message = message
        self.details = details or {}
        super().__init__(message)

# main.py
from fastapi import Request
from fastapi.responses import JSONResponse
from exceptions import ResourceNotFoundError, ValidationError, ConflictError
from constants.error_codes import ErrorCode

@app.exception_handler(ResourceNotFoundError)
async def resource_not_found_handler(request: Request, exc: ResourceNotFoundError):
    return JSONResponse(
        status_code=404,
        content={
            "success": False,
            "error": {
                "code": ErrorCode.NOT_FOUND,
                "message": str(exc),

async def validation_error_handler(request: Request, exc: ValidationError):

@app.exception_handler(ConflictError)
async def conflict_error_handler(request: Request, exc: ConflictError):

            "success": False,

## Review（自己採点チェックリスト）

### 必須項目

- [ ] ErrorDetail, ErrorResponse, SuccessResponse が定義されている
- [ ] ErrorCode 定数が定義されている
- [ ] カスタム例外クラスが作成されている
- [ ] グローバル例外ハンドラが実装されている
- [ ] サービス層がカスタム例外を使用している
- [ ] ADR-007 が作成されている
- [ ] 既存のテストが更新され、全て通る

### 加点項目

- [ ] エラーレスポンスに example がある（OpenAPI）
- [ ] エラーコードが体系的に整理されている
- [ ] success フィールドで成功/失敗を明確に区別している
- [ ] details に追加情報が含まれている

### よくある間違い

- カスタム例外を作らず、HTTPException だけ使う（統一できない）
- グローバルハンドラを追加しない（各エンドポイントで個別処理）
- ADR を書かない（設計判断が記録されない）
- 既存のテストを更新しない（テストが壊れる）
- エラーコードが乱立する（NOT_FOUND_USER, NOT_FOUND_PROJECT など細かすぎる）
