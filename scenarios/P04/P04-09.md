## Learn（概念理解）

### アーキテクチャドキュメントの重要性

コードベースが成長すると、**アーキテクチャが見えにくく**なる:

- 新メンバーがコード構造を理解するのに時間がかかる
- サービス層の責任範囲が不明確
- ディレクトリ構造の意図がわからない

**アーキテクチャ図**で視覚化する:
- システム全体の構造
- 各レイヤーの責任
- データの流れ

### C4 モデル

**4つのレベルでアーキテクチャを表現**:

1. **Context**: システム全体とユーザー・外部システムの関係
2. **Container**: アプリケーション、DB、外部サービス
3. **Component**: 各コンテナ内のコンポーネント（レイヤー）
4. **Code**: クラス図（詳細すぎるので通常は省略）

このシナリオでは **Component レベル**（3層アーキテクチャ）を図示:

```

│         Client (Browser)        │
└────────────┬────────────────────┘
             │ HTTP Request
             ▼
┌─────────────────────────────────┐
│      Presentation Layer         │
│      (FastAPI Routes)           │
│  - ルートハンドラ（薄い）        │
│  - リクエスト/レスポンス変換     │

│      Business Layer             │

│  - ビジネスロジック              │

             │ DB Query
             ▼
┌─────────────────────────────────┐

│      (SQLAlchemy Models)        │
│  - User, Project, Task          │

             │ SQL
```

## Mission（本番課題）

### 状況

P04 のリファクタリングが完了しました。TL から:

> 「コードはだいぶ整理されたけど、ドキュメントが古いままだね。README とアーキテクチャ図を更新してください。新メンバーが見ても理解できるように。」

### やること

1. **GitHub Issue を作成する**
   - タイトル: `docs: README とアーキテクチャ図を更新`

2. **ブランチを作成する**
   ```bash
   git checkout -b docs/proj04-architecture-update
   ```

3. **アーキテクチャ図を作成する**
   - ファイル: `docs/architecture.md`
   - 3層アーキテクチャの図（Component レベル）
   - 各レイヤーの責任を説明
   - データの流れを明記

4. **ディレクトリ構造を文書化する**
   - `docs/architecture.md` に追加
   - 各ディレクトリ・ファイルの役割を説明

5. **README を更新する**
   - アーキテクチャセクションを追加
   - 技術スタックを更新（mypy 追加など）
   - ADR へのリンクを追加
   - リファクタリング後の構造を反映

6. **ADR インデックスを作成する**
   - ファイル: `docs/adrs/README.md`
   - 全 ADR の一覧とリンク

7. **全テストが通ることを確認する**
   ```bash
   pytest tests/ -v
   ```

8. **PR を作成する**

### Git 操作の参考

```bash
git checkout -b docs/proj04-architecture-update

git add README.md docs/architecture.md docs/adrs/README.md

git commit -m "docs: update README and add architecture documentation

- Add architecture diagram (3-layer architecture)
- Document directory structure and responsibilities
- Update README with architecture section
- Add technology stack (mypy, etc.)
- Create ADR index with links
- Reflect refactored codebase structure

Closes #XX"

git push -u origin docs/proj04-architecture-update
```

```
## 各レイヤーの責任

### Presentation Layer
- HTTP リクエストの受け取り
- リクエストデータの Pydantic スキーマへの変換
- サービス層の呼び出し
- ビジネス例外を HTTP エラーに変換
- レスポンスの返却

**ルール**:
- ビジネスロジックを含まない（薄く保つ）
- サービス層への呼び出しのみ
- 5-10行程度

### Business Layer
- ビジネスロジックの実行
- バリデーション（ビジネスルール）
- データベース操作
- トランザクション管理
- 外部サービス呼び出し

**ルール**:
- HTTP に依存しない（HTTPException を投げない）
- ビジネス例外（ValueError など）を投げる
- テスト可能な設計

### Data Layer
- データベーススキーマの定義
- ORM マッピング
- リレーションの定義

**ルール**:
- ビジネスロジックを含まない
- SQLAlchemy のモデル定義のみ
```

## Review（自己採点チェックリスト）

### 必須項目

- [ ] docs/architecture.md が作成されている
- [ ] 3層アーキテクチャの図がある
- [ ] 各レイヤーの責任が説明されている
- [ ] ディレクトリ構造が文書化されている
- [ ] README が更新されている（アーキテクチャセクション）
- [ ] ADR インデックスが作成されている

### 加点項目

- [ ] 図が見やすい（矢印、レイヤー分けが明確）
- [ ] 新メンバーが読んで理解できる内容
- [ ] 技術スタックが最新（mypy など追加）
- [ ] 設計判断へのリンクがある

### よくある間違い

- 図が複雑すぎる（細かすぎて見にくい）
- ディレクトリ構造だけで、各ファイルの役割を説明していない
- README を更新しない（古い情報のまま）
- ADR へのリンクがない（設計判断が追えない）
