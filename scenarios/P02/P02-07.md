# P02-07: 未知フィールド送信が無視される

> **シナリオ**: フロントから「未知フィールド送ると無視される。拒否すべき?」
> **想定時間**: 4h
> **スキルタグ**: P4（コード品質）, P1（コードリーディング）, P3（テスト）

---

## Learn（概念理解）

### 未知フィールドの問題

デフォルトの Pydantic モデルでは、**定義されていないフィールドは無視される**:

```python
class UserCreate(BaseModel):
    name: str
    email: str

# "age" は定義にないが、エラーにならず無視される
data = UserCreate(name="Alice", email="alice@example.com", age=25)
# → name="Alice", email="alice@example.com"（age は消える）
```

これが問題になるケース:
- フロントがタイプミスしたフィールド（`emial` → 無視 → 原因不明のバグ）
- API のバージョン不一致（古いフィールドを送り続けても気づかない）
- 意図しないデータの送信（セキュリティリスク）

### `model_config = ConfigDict(extra="forbid")`

Pydantic v2 では `ConfigDict` を使って未知フィールドを拒否する:

```python
from pydantic import BaseModel, ConfigDict

# ❌ Bad: 未知フィールドを無視（デフォルト）
class UserCreate(BaseModel):
    name: str
    email: str

# ✅ Good: 未知フィールドを拒否
class UserCreate(BaseModel):
    model_config = ConfigDict(extra="forbid")

    name: str
    email: str
```

未知フィールドを送信した場合:
```json
// リクエスト
{"name": "Alice", "email": "alice@example.com", "age": 25}

// レスポンス: 422 Unprocessable Entity
{
  "detail": [
    {
      "type": "extra_forbidden",
      "loc": ["body", "age"],
      "msg": "Extra inputs are not permitted"
    }
  ]
}
```

### `extra` オプションの種類

| 設定 | 動作 | 使い所 |
|------|------|--------|
| `"ignore"` | 未知フィールドを無視（デフォルト） | 緩い API |
| `"forbid"` | 未知フィールドでエラー | 厳密な API（推奨） |
| `"allow"` | 未知フィールドを保持 | 柔軟な拡張が必要な場合 |

### ADR で方針を記録する

チーム全体で統一するため、ADR として記録する:

```markdown
# ADR: Pydantic スキーマで extra="forbid" を採用

## コンテキスト
未知フィールドが無視されるため、フロントのタイプミスに気づけない。

## 決定
全ての入力スキーマ（Create/Update/Patch）に extra="forbid" を設定する。

## 結果
- タイプミスが即座に 422 で検出される
- API 契約が厳密になる
- 将来のフィールド追加時は明示的にスキーマを更新する必要がある
```

## Practice（練習課題）

### 課題: 以下のスキーマに `extra="forbid"` を追加せよ

```python
class OrderCreate(BaseModel):
    product_id: int
    quantity: int

class OrderUpdate(BaseModel):
    quantity: int
```

**Q1.** 現状で `{"product_id": 1, "quantity": 5, "discount": 0.1}` を送信するとどうなるか？
**Q2.** `extra="forbid"` を追加後、同じリクエストを送信するとどうなるか？
**Q3.** `ConfigDict` を使って修正せよ。

### 模範解答

<details>
<summary>クリックで表示</summary>

**A1.** `discount` フィールドは無視され、`product_id=1, quantity=5` の注文が作成される。割引を意図していたのに適用されない。

**A2.** 422 Unprocessable Entity が返り、`"Extra inputs are not permitted"` のエラーメッセージで `discount` が未知フィールドであることが示される。

**A3.**
```python
from pydantic import BaseModel, ConfigDict

class OrderCreate(BaseModel):
    model_config = ConfigDict(extra="forbid")

    product_id: int
    quantity: int

class OrderUpdate(BaseModel):
    model_config = ConfigDict(extra="forbid")

    quantity: int
```

</details>

## Mission（本番課題）

### 状況

フロントエンドチームから相談がありました:

> 「`POST /users` で `{"name": "Alice", "emial": "alice@example.com"}` って送ったら（email のタイプミス）、エラーにならずに email が空のユーザーが作成されました。未知フィールドはエラーにすべきじゃないですか？」

### やること

1. **問題を再現する**
   ```bash
   curl -X POST http://localhost:8000/users \
     -H "Content-Type: application/json" \
     -d '{"name": "Alice", "emial": "alice@example.com"}'
   ```
   - エラーにならず、email が空（または必須なのに通る）ことを確認

2. **GitHub Issue を作成する**
   - タイトル: `fix: Pydantic スキーマで未知フィールドを拒否する`
   - 本文: 問題の説明、タイプミスが検出されない例

   <details>
   <summary>Issue の書き方例</summary>

   ```markdown
   ## 概要
   Pydantic スキーマで未知フィールドが無視され、タイプミスが検出されない。

   ## 再現手順
   1. POST /users に {"name": "Alice", "emial": "alice@example.com"} を送信（email の typo）
   2. 200/201 OK でユーザーが作成される
   3. email フィールドが空または None になっている

   ## 期待する動作
   422 Unprocessable Entity で拒否される。

   ## 実際の動作
   未知フィールドが無視され、typo が検出されない。

   ## 原因
   Pydantic スキーマに extra="forbid" が設定されていない。

   ## 影響範囲
   - 全入力スキーマ（UserCreate, UserUpdate, TeamCreate など）
   ```

   **ポイント**:
   - タイプミスの例を具体的に
   - extra="forbid" の重要性を強調
   </details>

3. **ブランチを作成する**
   ```bash
   git checkout -b fix/proj02-pydantic-extra-forbid
   ```

4. **全スキーマに `extra="forbid"` を追加する**
   - `UserCreate`
   - `UserUpdate`
   - `UserPatch`
   - `TeamCreate`
   - `TeamMemberCreate`
   - レスポンスモデル（`UserResponse` 等）には追加しない（出力は制限不要）

5. **テストを書く**
   - 未知フィールド付きで POST → 422 テスト
   - 正常なフィールドのみで POST → 200/201 テスト
   - 各スキーマで未知フィールドが拒否されるテスト

6. **ADR を作成する**
   - ファイル: `docs/adrs/ADR-xxx-pydantic-extra-forbid.md`
   - 方針と理由を記録

7. **PR を作成する**
   - Issue を参照する
   - 影響範囲（全入力スキーマ）を説明する

   <details>
   <summary>PR の書き方例</summary>

   ```markdown
   ## 概要
   全入力スキーマに extra="forbid" を追加し、未知フィールドを拒否するようにしました。

   Closes #XX

   ## 変更内容
   - 全入力スキーマに extra="forbid" を追加
   - テストを追加（未知フィールドで 422 が返る）
   - ADR を作成

   ## 理由
   タイプミス（email → emial など）が検出されず、データ不整合が発生していた。

   ## 影響範囲
   - UserCreate, UserUpdate, UserPatch
   - TeamCreate, TeamMemberCreate
   - レスポンスモデルは変更なし

   ## テスト
   - [ ] 未知フィールド付きで POST すると 422 が返る
   - [ ] 正常なフィールドのみで POST すると成功する
   - [ ] 既存テストが全て通る

   ## レビュー観点
   - [ ] 全入力スキーマに extra="forbid" があるか
   - [ ] レスポンスモデルには追加していないか
   ```

   **ポイント**:
   - タイプミス検出の重要性を強調
   - 影響範囲を明確に列挙
   </details>

### ヒント

<details>
<summary>どのスキーマに追加すべきかわからない場合</summary>

`main.py` の Pydantic モデルを確認:
- **追加する**: `UserCreate`, `UserUpdate`, `UserPatch`, `TeamCreate`, `TeamMemberCreate`（入力スキーマ）
- **追加しない**: `UserResponse`, `TeamResponse`, `TeamMemberResponse`（出力スキーマ）

入力スキーマだけに追加する理由: 出力はサーバーが制御するので制限不要。

</details>

<details>
<summary>テストの書き方がわからない場合</summary>

```python
def test_create_user_extra_field_rejected():
    """未知フィールドが拒否される"""
    response = client.post("/users", json={
        "name": "Alice",
        "email": "alice@example.com",
        "age": 25,  # 未知フィールド
    })
    assert response.status_code == 422
    assert "extra_forbidden" in response.text
```

</details>

### よくある間違い

- レスポンスモデルにも `extra="forbid"` を追加してしまう
- `class Config: extra = "forbid"` と書く（Pydantic v1 の書き方。v2 では `model_config = ConfigDict(extra="forbid")`）
- 既存テストが未知フィールドを送信していて壊れる（テストも修正が必要）
- `UserPatch` に `extra="forbid"` を追加し忘れる
