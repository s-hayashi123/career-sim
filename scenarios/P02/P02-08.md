## Learn（概念理解）

### 外部キー制約（Foreign Key）とは

あるテーブルのカラムが、別のテーブルの主キーを参照する制約:

```
users テーブル          team_members テーブル
+----+--------+        +----+---------+---------+
| id | name   |        | id | team_id | user_id |
+----+--------+        +----+---------+---------+
| 1  | Alice  |   ←──  | 1  | 10      | 1       |
| 2  | Bob    |   ←──  | 2  | 10      | 2       |
+----+--------+        +----+---------+---------+
```

`team_members.user_id` が `users.id` を参照している。Alice を削除しようとすると、team_members にまだ Alice を参照するレコードがあるため **外部キー制約違反** でエラーになる。

### ON DELETE オプション

外部キー制約に `ON DELETE` を設定することで、参照先が削除されたときの動作を指定できる:

| オプション | 動作 | 使い所 |
|-----------|------|--------|
| `RESTRICT` (デフォルト) | 削除を拒否（エラー） | データ保全が最優先 |
| `CASCADE` | 参照しているレコードも自動削除 | 親に完全依存するデータ |
| `SET NULL` | 外部キーを NULL に設定 | 参照が切れても残したいデータ |

```python
# CASCADE: ユーザー削除時に team_members も自動削除
class TeamMember(Base):
    user_id = Column(Integer, ForeignKey("users.id", ondelete="CASCADE"))

# SET NULL: ユーザー削除時に user_id を NULL にする
class TeamMember(Base):
    user_id = Column(Integer, ForeignKey("users.id", ondelete="SET NULL"), nullable=True)
```

### 削除前の依存チェック

`ON DELETE` を設定する代わりに、**アプリケーション側で削除前にチェック**する方法もある:

```python
# ❌ Bad: いきなり削除 → 外部キー制約違反で IntegrityError → 500
@app.delete("/users/{user_id}")
def delete_user(user_id: int, db: Session = Depends(get_db)):
    user = db.query(User).filter(User.id == user_id).first()
    db.delete(user)
    db.commit()  # IntegrityError!

# ✅ Good: 依存チェック → エラーメッセージで案内
@app.delete("/users/{user_id}")
def delete_user(user_id: int, db: Session = Depends(get_db)):
    user = db.query(User).filter(User.id == user_id).first()
    if user is None:
        raise HTTPException(status_code=404, detail="ユーザーが見つかりません")

    # 依存チェック
    member_count = db.query(TeamMember).filter(TeamMember.user_id == user_id).count()
    if member_count > 0:
        raise HTTPException(
            status_code=409,
            detail="このユーザーはチームに所属しているため削除できません"
        )

    db.delete(user)
    db.commit()
    return {"detail": "User deleted"}
```

### 論理削除 vs 物理削除

| 方式 | 動作 | メリット | デメリット |
|------|------|---------|----------|
| 物理削除 | `DELETE FROM users WHERE id=1` | シンプル | 復元不可 |
| 論理削除 | `UPDATE users SET deleted_at=NOW()` | 復元可能、監査対応 | クエリに条件追加が必要 |

ADR で方針を決めておくのが重要。

## Practice（練習課題）

### 課題: 以下のコードの問題点を見つけて修正せよ

```python
class Comment(Base):
    __tablename__ = "comments"
    id = Column(Integer, primary_key=True)
    post_id = Column(Integer, ForeignKey("posts.id"))
    body = Column(String)

@app.delete("/posts/{post_id}")
def delete_post(post_id: int, db: Session = Depends(get_db)):
    post = db.query(Post).filter(Post.id == post_id).first()
    db.delete(post)
    db.commit()
    return {"detail": "Post deleted"}
```

**Q1.** コメントがある投稿を削除しようとすると何が起こるか？
**Q2.** 解決方法を3つ挙げよ（それぞれのメリット・デメリット）。
**Q3.** 依存チェックによる解決方法でコードを書け。

### 模範解答

<details>
<summary>クリックで表示</summary>

**A1.** 外部キー制約違反で `IntegrityError` が発生し、500 Internal Server Error が返る。

**A2.**
1. **CASCADE** — `ForeignKey("posts.id", ondelete="CASCADE")` でコメントも自動削除。簡単だが、意図しない大量削除のリスクあり。
2. **依存チェック** — 削除前にコメントの有無を確認し、あればエラー。安全だが実装が増える。
3. **論理削除** — `deleted_at` フラグで論理削除。復元可能だが、全クエリに条件追加が必要。

**A3.**
```python
from fastapi import HTTPException

@app.delete("/posts/{post_id}")
def delete_post(post_id: int, db: Session = Depends(get_db)):
    post = db.query(Post).filter(Post.id == post_id).first()
    if post is None:
        raise HTTPException(status_code=404, detail="投稿が見つかりません")

    comment_count = db.query(Comment).filter(Comment.post_id == post_id).count()
    if comment_count > 0:
        raise HTTPException(
            status_code=409,
            detail=f"この投稿には {comment_count} 件のコメントがあるため削除できません"
        )

    db.delete(post)
    db.commit()
    return {"detail": "Post deleted"}
```

</details>

## Mission（本番課題）

### 状況

QA チームから報告がありました:

> 「チームに所属しているユーザーを `DELETE /users/{id}` で削除すると 500 になります。`team_members` テーブルに外部キーで参照されているのが原因っぽいです。ユーザーを安全に削除できるようにしてください。」

### やること

1. **バグを再現する**
   ```bash
   # ユーザー作成
   curl -X POST http://localhost:8000/users \
     -H "Content-Type: application/json" \
     -d '{"name": "Alice", "email": "alice@example.com"}'

   # チーム作成
   curl -X POST http://localhost:8000/teams \
     -H "Content-Type: application/json" \
     -d '{"name": "Engineering"}'

   # チームに Alice を追加
   curl -X POST http://localhost:8000/teams/1/members \
     -H "Content-Type: application/json" \
     -d '{"user_id": 1}'

   # Alice を削除 → 500
   curl -X DELETE http://localhost:8000/users/1
   ```

2. **GitHub Issue を作成する**
   - タイトル: `fix: チームメンバーのユーザー削除で500（外部キー制約違反）`
   - 本文: 再現手順、原因（ON DELETE 未設定 + 依存チェックなし）

   <details>
   <summary>Issue の書き方例</summary>

   ```markdown
   ## 概要
   チームに所属しているユーザーを削除すると500エラーが発生する。

   ## 再現手順
   1. ユーザーAlice (id=1) を作成
   2. チーム Project A を作成
   3. Alice を Project A に追加
   4. DELETE /users/1 を実行
   5. 500 Internal Server Error が返る

   ## 期待する動作
   - 方針 A: 409 Conflict で拒否（チームに所属中のため削除不可）
   - 方針 B: 200 OK で削除（関連する team_members も削除）
   - 方針 C: 200 OK で論理削除

   ## 実際の動作
   500 Internal Server Error（外部キー制約違反）

   ## 原因
   - team_members テーブルの user_id に外部キー制約があるが ON DELETE が未設定
   - 削除前の依存チェックもない

   ## 影響範囲
   - チーム所属ユーザーの削除時
   ```

   **ポイント**:
   - 外部キー制約の問題を明確に
   - 複数の方針を提示
   </details>

3. **ブランチを作成する**
   ```bash
   git checkout -b fix/proj02-delete-user-foreign-key
   ```

4. **修正する**（以下のいずれかの方針）
   - **方針 A**: 削除前に依存チェック → 409 Conflict で拒否
   - **方針 B**: `ON DELETE CASCADE` で関連レコードも削除
   - **方針 C**: 論理削除に変更
   - どの方針を選んだかを ADR に記録する

5. **存在チェックも追加する**
   - 存在しない ID で DELETE → 404

6. **テストを書く**
   - チーム所属ユーザーの削除 → 方針に応じた結果のテスト

   - 存在しない ID で DELETE → 404 テスト

7. **ADR-003 を作成する**
   - ファイル: `docs/adrs/ADR-003-deletion-policy.md`
   - 物理削除 vs 論理削除の方針、外部キー制約の扱い

8. **PR を作成する**
   - Issue を参照する
   - 選択した方針とその理由を説明する

   <summary>PR の書き方例</summary>

   ## 原因
   team_members テーブルの外部キー制約により、チーム所属ユーザーの削除時に IntegrityError が発生していた。

### ヒント

<details>
<summary>どの方針を選ぶか迷う場合</summary>

初学者には**方針 A（依存チェック + 409 で拒否）**がおすすめ:
- 実装がシンプル
- 意図しないデータ削除のリスクがない
- ユーザーに明確なエラーメッセージを返せる

将来的に「チームから外してから削除」のフローをフロントに実装すればよい。

</details>

<details>
<summary>テストの書き方がわからない場合</summary>

```python
def test_delete_user_with_team_membership():
    """チームに所属するユーザーの削除は 409"""
    # ユーザー作成
    client.post("/users", json={"name": "Alice", "email": "alice@example.com"})
    # チーム作成
    client.post("/teams", json={"name": "Engineering"})
    # チームに追加
    client.post("/teams/1/members", json={"user_id": 1})
    # 削除 → 409
    response = client.delete("/users/1")
    assert response.status_code == 409
```

### よくある間違い

- 外部キー制約違反の `IntegrityError` を汎用的にキャッチして原因を隠す
- CASCADE を安易に選んで意図しない大量削除を許可する
- 存在チェックを忘れる（存在しない ID で DELETE → 別のエラー）
- ADR を書かずに実装だけする（方針の根拠が残らない）
- 論理削除を選んだのに全クエリに `WHERE deleted_at IS NULL` を追加し忘れる
