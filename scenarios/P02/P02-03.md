## Learn（概念理解）

### 更新時のユニーク検証が必要な理由

「作成時にはメール重複をチェックしているのに、更新時にはチェックしていない」というバグは実務でよく発生する:

```
ユーザー A: alice@example.com
ユーザー B: bob@example.com

PUT /users/2 {"name": "Bob", "email": "alice@example.com"}
→ 200 OK（Alice のメールに変更できてしまう！）
```

### 更新時ユニーク検証の「自分自身を除外」パターン

更新時の重複チェックでは、**自分自身を除外**する必要がある。そうしないと、メールを変更しない更新でも「重複」と判定されてしまう:

```python
# ❌ Bad: 自分自身を除外していない
existing = db.query(User).filter(User.email == user_data.email).first()
if existing:
    raise HTTPException(409)  # 自分のメールを変えない更新でも 409 になる！

# ✅ Good: 自分自身を除外する
existing = db.query(User).filter(
    User.email == user_data.email,
    User.id != user_id  # 自分自身は除外
).first()
if existing:
    raise HTTPException(status_code=409, detail="このメールアドレスは既に使用されています")
```

### 存在チェックも忘れずに

更新対象のユーザーが存在しない場合も考慮が必要:

```python
# ❌ Bad: 存在チェックなし → None に対して属性代入で AttributeError
user = db.query(User).filter(User.id == user_id).first()
user.name = user_data.name  # user が None なら AttributeError → 500!

# ✅ Good: 存在チェック + 重複チェック
user = db.query(User).filter(User.id == user_id).first()
if user is None:
    raise HTTPException(status_code=404, detail="ユーザーが見つかりません")

existing = db.query(User).filter(
    User.email == user_data.email,
    User.id != user_id
).first()
if existing:
    raise HTTPException(status_code=409, detail="このメールアドレスは既に使用されています")
```

### 409 Conflict vs 422 Unprocessable Entity

| コード | 意味 | 更新時の使い所 |
|-------|------|--------------|
| 404 | Not Found | 更新対象が存在しない |
| 409 | Conflict | 他のリソースとの競合（メール重複など） |
| 422 | Unprocessable Entity | 入力形式が不正（メール形式違反など） |

**409 は「データとしては正しいが、既存リソースと競合する」**場合に使う。

## Practice（練習課題）

### 課題: 以下の更新エンドポイントに重複チェックを追加せよ

```python
class Product(Base):
    __tablename__ = "products"
    id = Column(Integer, primary_key=True)
    sku = Column(String, nullable=False, unique=True)
    name = Column(String, nullable=False)

@app.put("/products/{product_id}")
def update_product(product_id: int, data: ProductUpdate, db: Session = Depends(get_db)):
    product = db.query(Product).filter(Product.id == product_id).first()
    product.sku = data.sku
    product.name = data.name
    db.commit()
    db.refresh(product)
    return product
```

**Q1.** このコードには問題が2つある。それぞれ何か？
**Q2.** `PUT /products/999`（存在しない ID）を呼ぶと何が起こるか？
**Q3.** 存在チェックと SKU 重複チェック（自分自身除外）を追加せよ。

### 模範解答

<details>
<summary>クリックで表示</summary>

**A1.**
1. `product` が `None`（存在しない ID）のとき、`product.sku = ...` で `AttributeError` → 500
2. 他の商品と同じ SKU に変更できてしまう（DB の UNIQUE 制約があれば `IntegrityError` → 500）

**A2.** `AttributeError: 'NoneType' object has no attribute 'sku'` が発生し、500 Internal Server Error が返る。

**A3.**
```python
from fastapi import HTTPException

@app.put("/products/{product_id}")
def update_product(product_id: int, data: ProductUpdate, db: Session = Depends(get_db)):
    product = db.query(Product).filter(Product.id == product_id).first()
    if product is None:
        raise HTTPException(status_code=404, detail="商品が見つかりません")

    existing = db.query(Product).filter(
        Product.sku == data.sku,
        Product.id != product_id
    ).first()
    if existing:
        raise HTTPException(status_code=409, detail="この SKU は既に使用されています")

    product.sku = data.sku
    product.name = data.name
    db.commit()
    db.refresh(product)
    return product
```

</details>

## Mission（本番課題）

### 状況

QA チームから Slack で報告がありました:

> 「`PUT /users/{id}` で既存ユーザーのメールアドレスに変更できちゃいます。Alice のメールを Bob と同じにできます。これ、メールはユニークじゃないとダメですよね？あと、存在しない ID で PUT すると 500 になります。」

### やること

1. **バグを再現する**
   - まずユーザーを2人作成する
   ```bash
   curl -X POST http://localhost:8000/users \
     -H "Content-Type: application/json" \
     -d '{"name": "Alice", "email": "alice@example.com"}'

   curl -X POST http://localhost:8000/users \
     -H "Content-Type: application/json" \
     -d '{"name": "Bob", "email": "bob@example.com"}'
   ```
   - Bob のメールを Alice のものに変更してみる
   ```bash
   curl -X PUT http://localhost:8000/users/2 \
     -H "Content-Type: application/json" \
     -d '{"name": "Bob", "email": "alice@example.com"}'
   ```
   - 200 OK で変更できることを確認（=バグ）
   - 存在しない ID で PUT して 500 を確認
   ```bash
   curl -X PUT http://localhost:8000/users/999 \
     -H "Content-Type: application/json" \
     -d '{"name": "Ghost", "email": "ghost@example.com"}'
   ```

2. **GitHub Issue を作成する**
   - タイトル: `fix: PUT /users/{id} でメール重複チェックと存在チェックを追加`
   - 本文: 2つのバグ（重複メール許可 + 存在しない ID で 500）の再現手順を記載

3. **原因を特定する**
   - `main.py` の `update_user` 関数（184-194行目）を読む
   - 存在チェックなし + メール重複チェックなし

4. **ブランチを作成する**
   ```bash
   git checkout -b fix/proj02-update-user-validation
   ```

5. **修正する**
   - ユーザー存在チェック（`None` → 404）を追加
   - メール重複チェック（自分自身を除外するクエリ → 409）を追加

6. **テストを書く**
   - 存在しない ID で PUT → 404 テスト
   - 他ユーザーのメールに変更 → 409 テスト
   - 自分のメールを変更しない更新 → 200 テスト（自分自身除外の確認）
   - 正常なメール変更 → 200 テスト

7. **PR を作成する**
   - Issue を参照する
   - 2つのバグの原因と修正内容を説明する
