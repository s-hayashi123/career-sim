## Learn（概念理解）

### なぜ DB 制約が必要なのか

アプリケーションコードだけでユニーク性を保証しようとすると、**同時リクエスト**で破綻する:

```
リクエスト A: SELECT → メール未登録 → INSERT  ← ここで OK と判断
リクエスト B: SELECT → メール未登録 → INSERT  ← A の INSERT がまだ commit 前なので OK と判断
→ 同じメールのユーザーが2件できる（レースコンディション）
```

**DB の UNIQUE 制約**は、INSERT/UPDATE の瞬間にデータベースレベルでチェックするため、同時リクエストでも絶対に重複しない。

### UNIQUE 制約とは

```python
# ❌ Bad: UNIQUE 制約なし — アプリ側だけで重複チェック
class User(Base):
    __tablename__ = "users"
    email = Column(String, nullable=False)

# ✅ Good: DB レベルで UNIQUE 制約
class User(Base):
    __tablename__ = "users"
    email = Column(String, nullable=False, unique=True)
```

既存のテーブルに後から UNIQUE 制約を追加するには、Alembic マイグレーションを使う:

```python
# alembic/versions/xxxx_add_unique_email.py
def upgrade():
    op.create_unique_constraint("uq_users_email", "users", ["email"])

def downgrade():
    op.drop_constraint("uq_users_email", "users")
```

### IntegrityError のハンドリング

UNIQUE 制約に違反すると SQLAlchemy は `IntegrityError` を送出する。これを**キャッチしないと 500 エラー**になる:

```python
from sqlalchemy.exc import IntegrityError
from fastapi import HTTPException

# ❌ Bad: IntegrityError を無視 → 500 Internal Server Error
@app.post("/users")
def create_user(user: UserCreate, db: Session = Depends(get_db)):
    db_user = User(name=user.name, email=user.email)
    db.add(db_user)
    db.commit()  # UNIQUE 違反で IntegrityError → 500!
    return db_user

# ✅ Good: IntegrityError をキャッチして 409 を返す
@app.post("/users")
def create_user(user: UserCreate, db: Session = Depends(get_db)):
    db_user = User(name=user.name, email=user.email)
    db.add(db_user)
    try:
        db.commit()
    except IntegrityError:
        db.rollback()
        raise HTTPException(status_code=409, detail="このメールアドレスは既に使用されています")
    db.refresh(db_user)
    return db_user
```

### HTTP 409 Conflict

| コード | 意味 | 使い所 |
|-------|------|--------|
| 201 | Created | リソース作成成功 |
| 409 | Conflict | リソースが既存のものと競合（重複など） |
| 422 | Unprocessable Entity | バリデーションエラー（形式不正） |
| 500 | Internal Server Error | サーバー側のバグ |

**409 はクライアントに「同じリソースが既に存在する」ことを伝える**。422 は「入力形式が不正」なので意味が違う。

## Practice（練習課題）

### 課題: 以下のコードの問題点を見つけて修正せよ

```python
class Product(Base):
    __tablename__ = "products"
    id = Column(Integer, primary_key=True)
    sku = Column(String, nullable=False)  # SKU は本来一意であるべき
    name = Column(String, nullable=False)

@app.post("/products")
def create_product(product: ProductCreate, db: Session = Depends(get_db)):
    db_product = Product(sku=product.sku, name=product.name)
    db.add(db_product)
    db.commit()
    db.refresh(db_product)
    return db_product
```

**Q1.** このコードにはどんな問題があるか？（2つ）
**Q2.** `sku` が重複する Product を同時に作成した場合、何が起こるか？
**Q3.** モデルとエンドポイントの両方を修正せよ。

### 模範解答

<details>
<summary>クリックで表示</summary>

**A1.**
1. `sku` カラムに `unique=True` がないため、同じ SKU の商品が複数作成できる
2. `IntegrityError` のハンドリングがないため、制約を追加しても違反時に 500 になる

**A2.** 同じ SKU を持つ Product が2件作成される。データの整合性が壊れる。

**A3.**
```python
from sqlalchemy.exc import IntegrityError
from fastapi import HTTPException

class Product(Base):
    __tablename__ = "products"
    id = Column(Integer, primary_key=True)
    sku = Column(String, nullable=False, unique=True)  # UNIQUE 追加
    name = Column(String, nullable=False)

@app.post("/products")
def create_product(product: ProductCreate, db: Session = Depends(get_db)):
    db_product = Product(sku=product.sku, name=product.name)
    db.add(db_product)
    try:
        db.commit()
    except IntegrityError:
        db.rollback()
        raise HTTPException(status_code=409, detail="この SKU は既に使用されています")
    db.refresh(db_product)
    return db_product
```

</details>

## Mission（本番課題）

### 状況

QA チームの Slack チャンネルに以下の報告が上がりました:

> 「ユーザー登録の負荷テストで、同じメールアドレスのユーザーが2件作られました。`POST /users` を同時に叩くと再現します。メールアドレスはユニークであるべきですよね？」

### やること

1. **バグを再現する**
   - 同じメールアドレスで2回 `POST /users` を叩く
   ```bash
   curl -X POST http://localhost:8000/users \
     -H "Content-Type: application/json" \
     -d '{"name": "Alice", "email": "alice@example.com"}'

   curl -X POST http://localhost:8000/users \
     -H "Content-Type: application/json" \
     -d '{"name": "Bob", "email": "alice@example.com"}'
   ```
   - 2件とも 200 OK で作成されることを確認（=バグ）

2. **GitHub Issue を作成する**
   - タイトル: `fix: POST /users で同じメールアドレスのユーザーが作成できる`
   - 本文: 再現手順・期待する動作（409 Conflict）・実際の動作（200 OK で重複作成）

   <details>
   <summary>Issue の書き方例</summary>

   ```markdown
   ## 概要
   POST /users で同じメールアドレスのユーザーが複数作成できてしまう。

   ## 再現手順
   1. POST /users に {"name": "Alice", "email": "alice@example.com"} を送信
   2. 200 OK でユーザーが作成される
   3. 再度 POST /users に {"name": "Bob", "email": "alice@example.com"} を送信
   4. 200 OK で同じメールアドレスのユーザーが作成される

   ## 期待する動作
   2回目のリクエストは 409 Conflict で拒否される。

   ## 実際の動作
   200 OK で重複したユーザーが作成される。

   ## 原因
   - users.email カラムに UNIQUE 制約がない
   - アプリ側でレースコンディションが発生している

   ## 影響範囲
   - 同時ユーザー登録時にデータ重複が発生
   - データの整合性が壊れる

   ## 環境
   - ローカル開発環境
   ```

   **ポイント**:
   - 再現手順を詳細に記載
   - 期待する動作と実際の動作を対比
   - 原因（DB制約不足）を明記
   </details>

3. **原因を特定する**
   - `main.py` の `User` モデル（29行目）を確認 → `email` に `unique=True` がない
   - `create_user` 関数（152-161行目）を確認 → `IntegrityError` のハンドリングがない

4. **ブランチを作成する**
   ```bash
   git checkout -b fix/proj02-unique-email-constraint
   ```

5. **DB マイグレーションを作成する**
   - `User` モデルの `email` カラムに `unique=True` を追加
   - Alembic マイグレーションを生成・適用

6. **IntegrityError ハンドリングを追加する**
   - `create_user` で `IntegrityError` をキャッチ
   - 409 Conflict を返す（エラーメッセージ付き）
   - `db.rollback()` を忘れないこと

7. **テストを書く**
   - 同じメールで2回 POST → 2回目が 409 を返すテスト
   - 異なるメールで POST → 正常に作成されるテスト
   - メール重複時のエラーメッセージが適切であるテスト

8. **PR を作成する**
   - Issue を参照する
   - 原因（UNIQUE 制約なし + エラーハンドリングなし）と修正内容を説明する

   <details>
   <summary>PR の書き方例</summary>

   - Alembic マイグレーションを作成・適用

   - アプリ側のチェックだけでは同時リクエストでレースコンディションが発生する

   ## 修正内容
   - DB レベルで UNIQUE 制約を追加（レースコンディション対策）

### ヒント

<details>
<summary>Alembic マイグレーションの作り方がわからない場合</summary>

```bash
# モデルを修正した後
alembic revision --autogenerate -m "add unique constraint to users.email"
alembic upgrade head
```

もし既に重複データがある場合は、マイグレーション前に重複を解消する必要がある。

</details>

<details>
<summary>IntegrityError のキャッチ方法がわからない場合</summary>

```python
from sqlalchemy.exc import IntegrityError

try:
    db.commit()
except IntegrityError:
    db.rollback()
    raise HTTPException(status_code=409, detail="メールアドレスが重複しています")
```

`db.rollback()` を呼ばないと、以降の DB 操作がすべて失敗するので注意。

</details>

### 加点項目

- [ ] エラーメッセージが具体的（「メールアドレスが重複しています」など）
- [ ] 正常系（異なるメールで作成成功）のテストもある
- [ ] PR 説明に「なぜアプリ側チェックだけでは不十分か（レースコンディション）」が記載されている
- [ ] 既存の重複データを解消するマイグレーションも考慮した

### よくある間違い

- DB 制約を追加せず、アプリ側の `SELECT` チェックだけで済ませる（レースコンディションが残る）
- `IntegrityError` をキャッチしたが `db.rollback()` を忘れる（以降のリクエストが壊れる）
- 409 ではなく 400 や 422 を返す（意味が異なる）
- テストで重複チェックの順序を考慮していない（1件目の作成を前提としていない）
