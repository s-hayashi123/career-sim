## Learn（概念理解）

### スタックトレース漏洩とは

未ハンドルの例外が発生すると、FastAPI はデフォルトで **スタックトレース（内部のコードパス）** をレスポンスに含める。これはセキュリティリスク:

```
# 攻撃者に見えてしまう情報
Traceback (most recent call last):
  File "/app/main.py", line 178, in get_user
    print(f"Found user: {user.name}")
AttributeError: 'NoneType' object has no attribute 'name'
```

漏洩する情報:
- ファイルパス（ディレクトリ構成がわかる）
- ソースコードの一部（ロジックが推測できる）
- 使用ライブラリとバージョン
- DB の構造やクエリ

### グローバル例外ハンドラ

全エンドポイントを個別に `try/except` するのではなく、**アプリ全体で未ハンドル例外をキャッチ**する:

```python
# ❌ Bad: 各エンドポイントに try/except（冗長で漏れが出る）
@app.get("/users/{user_id}")
def get_user(user_id: int):
    try:
        ...
    except Exception:
        return JSONResponse(status_code=500, content={"detail": "Internal Server Error"})

# ✅ Good: グローバル例外ハンドラで一括キャッチ
from fastapi import Request
from fastapi.responses import JSONResponse
import logging

logger = logging.getLogger(__name__)

@app.exception_handler(Exception)
async def global_exception_handler(request: Request, exc: Exception):
    logger.error("Unhandled exception: %s", exc, exc_info=True)
    return JSONResponse(
        status_code=500,
        content={"detail": "Internal Server Error"},
    )
```

### 例外ハンドラの優先順位

FastAPI の例外ハンドラは **具体的なものが優先される**:

```python
# 1. HTTPException は FastAPI が自動処理（404, 422 など）
# 2. 特定の例外タイプ用ハンドラ
@app.exception_handler(IntegrityError)
async def integrity_error_handler(request, exc):
    return JSONResponse(status_code=409, content={"detail": "データが競合しています"})

# 3. グローバルハンドラ（上記でキャッチされなかったもの全て）
@app.exception_handler(Exception)
async def global_exception_handler(request, exc):
    logger.error("Unhandled exception: %s", exc, exc_info=True)
    return JSONResponse(status_code=500, content={"detail": "Internal Server Error"})
```

### ADR（Architecture Decision Record）とは

重要な設計判断を記録するドキュメント:

```markdown
# ADR-002: グローバル例外ハンドラの導入

## ステータス
承認

## コンテキスト
未ハンドル例外でスタックトレースがレスポンスに含まれる。

## 決定
FastAPI の exception_handler でグローバルにキャッチし、
ログには詳細を記録、レスポンスには汎用メッセージのみ返す。

## 結果
- セキュリティリスクの低減
- 統一されたエラーレスポンス形式
```

## Practice（練習課題）

### 課題: 以下のアプリにグローバル例外ハンドラを追加せよ

```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/divide/{a}/{b}")
def divide(a: int, b: int):
    return {"result": a / b}  # b=0 で ZeroDivisionError → スタックトレース漏洩
```

**Q1.** `GET /divide/10/0` を叩くとどうなるか？
**Q2.** グローバル例外ハンドラを追加して、安全なエラーレスポンスを返すようにせよ。
**Q3.** ログにはエラーの詳細を残す必要がある。どう実装するか？

### 模範解答

<details>
<summary>クリックで表示</summary>

**A1.** `ZeroDivisionError: division by zero` のスタックトレースがレスポンスに含まれて返る（500）。

**A2. / A3.**
```python
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse
import logging

logger = logging.getLogger(__name__)

app = FastAPI()

@app.exception_handler(Exception)
async def global_exception_handler(request: Request, exc: Exception):
    logger.error("Unhandled exception on %s: %s", request.url, exc, exc_info=True)
    return JSONResponse(
        status_code=500,
        content={"detail": "Internal Server Error"},
    )

@app.get("/divide/{a}/{b}")
def divide(a: int, b: int):
    return {"result": a / b}
```

ログには `exc_info=True` でスタックトレースを記録するが、レスポンスには `"Internal Server Error"` だけ返す。

</details>

## Mission（本番課題）

### 状況

モニタリングチームから Slack で連絡がありました:

> 「本番の API レスポンスにスタックトレースが含まれているリクエストを検知しました。セキュリティ上の懸念があります。未ハンドルの例外が発生したとき、内部情報をレスポンスに含めないようにしてください。ログにはちゃんと残してください。」

### やること

1. **問題を再現する**
   - 存在しないユーザー ID でリクエストしてスタックトレースが返ることを確認
   ```bash
   curl http://localhost:8000/users/999
   ```
   - レスポンスにコードパスや `AttributeError` 等が含まれていることを確認

2. **GitHub Issue を作成する**
   - タイトル: `fix: 未ハンドル例外でスタックトレースが漏洩する`
   - 本文: セキュリティリスクの説明、再現手順

   <details>
   <summary>Issue の書き方例</summary>

   ```markdown
   ## 概要
   未ハンドル例外が発生すると、スタックトレースがレスポンスに含まれ漏洩する。

   ## 再現手順
   1. GET /users/999 を実行（存在しない ID）
   2. 500 Internal Server Error が返る
   3. レスポンスにスタックトレースやコードパスが含まれる

   ## 期待する動作
   スタックトレースは含まず、汎用的なエラーメッセージのみ返す。

   ## 実際の動作
   ```json
   {
     "detail": "AttributeError: 'NoneType' object has no attribute 'name'",
     "traceback": [...]
   }
   ```

   ## セキュリティリスク
   - コードの内部構造が漏洩
   - 攻撃者が脆弱性を特定しやすくなる

   ## 影響範囲
   - 全ての未ハンドル例外
   ```

   **ポイント**:
   - セキュリティリスクを明確に
   - スタックトレースの例を含める
   </details>

3. **ブランチを作成する**
   ```bash
   git checkout -b fix/proj02-global-exception-handler
   ```

4. **グローバル例外ハンドラを実装する**
   - `@app.exception_handler(Exception)` を追加
   - レスポンス: `{"detail": "Internal Server Error"}` のみ
   - ログ: `logger.error()` で `exc_info=True` 付きの詳細を記録

5. **テストを書く**
   - 未ハンドル例外で 500 が返ること
   - レスポンスにスタックトレースが含まれないこと
   - レスポンスが `{"detail": "Internal Server Error"}` 形式であること

6. **ADR-002 を作成する**
   - ファイル: `docs/adrs/ADR-002-global-exception-handler.md`
   - コンテキスト・決定・結果を記載

7. **PR を作成する**
   - Issue を参照する
   - セキュリティ対策としての説明を含める

   <details>
   <summary>PR の書き方例</summary>

   ```markdown
   ## 概要
   グローバル例外ハンドラを実装し、スタックトレース漏洩を防止しました。

   Closes #XX

   ## 変更内容
   - @app.exception_handler(Exception) を追加
   - レスポンスは汎用的なメッセージのみ
   - 詳細ログは logger.error() で記録
   - ADR-002 を作成

   ## セキュリティ対策
   未ハンドル例外発生時にスタックトレースを返さず、汎用的なエラーメッセージのみを返す。

   ## Before
   ```json
   {"detail": "AttributeError: ...", "traceback": [...]}
   ```

   ## After
   ```json
   {"detail": "Internal Server Error"}
   ```

   ## テスト
   - [ ] 未ハンドル例外で 500 が返る
   - [ ] レスポンスにスタックトレースが含まれない
   - [ ] 詳細ログが記録される

   ## レビュー観点
   - [ ] グローバル例外ハンドラが適切か
   - [ ] ログに十分な情報が記録されているか
   ```

   **ポイント**:
   - セキュリティ対策であることを強調
   - Before/After で改善を明示
   </details>

## Review（自己採点チェックリスト）

### 必須項目

- [ ] グローバル例外ハンドラ `@app.exception_handler(Exception)` が実装されている
- [ ] レスポンスに `{"detail": "Internal Server Error"}` のみ返している
- [ ] レスポンスにスタックトレースが含まれない
- [ ] ログに `exc_info=True` でエラー詳細が記録される
- [ ] テストが追加されている
- [ ] ADR-002 が作成されている
- [ ] GitHub Issue と PR が作成されている

### 加点項目

- [ ] `HTTPException` は通常通り処理される（404 等が 500 にならない）
- [ ] リクエストの URL やメソッドがログに含まれている
- [ ] ADR にセキュリティリスクの説明がある
- [ ] 環境変数で debug モード切替を検討した

### よくある間違い

- `HTTPException` まで global handler でキャッチしてしまう（FastAPI が先に処理するので通常は問題ないが、意図を理解しておく）
- ログに `exc_info=True` を付け忘れる（スタックトレースがログにも残らない）
- レスポンスにエラーの種類（`AttributeError` など）を含めてしまう
- ADR を書き忘れる
