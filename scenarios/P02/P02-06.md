## Learn（概念理解）

### 回帰テスト（Regression Test）とは

**修正したバグが再発しないことを保証するテスト**。コードの変更で既存の動作が壊れていないかを検出する。

```
バグ修正の流れ:
  1. バグ報告 → 2. 再現 → 3. 修正 → 4. 回帰テスト追加 → 5. PR
                                         ↑ これがないと再発リスク
```

### テストの分類: Unit vs Integration

| 種類 | テスト対象 | 速度 | 例 |
|------|----------|------|-----|
| Unit テスト | 関数単体 | 高速 | バリデーション関数、ユーティリティ |
| Integration テスト | API エンドポイント全体 | 中速 | `client.get("/users")` → DB → レスポンス |

```
tests/
├── unit/
│   ├── test_validators.py      # Pydantic バリデーション
│   └── test_utils.py           # ユーティリティ関数
├── integration/
│   ├── test_users_api.py       # ユーザー API のエンドポイント
│   └── test_teams_api.py       # チーム API のエンドポイント
└── conftest.py                 # 共通フィクスチャ
```

### pytest のディレクトリ構成ベストプラクティス

```python
# tests/conftest.py — 共通設定
import pytest
from fastapi.testclient import TestClient
from main import app

@pytest.fixture
def client():
    return TestClient(app)

@pytest.fixture
def sample_user(client):
    """テスト用ユーザーを作成するフィクスチャ"""
    response = client.post("/users", json={"name": "Test User", "email": "test@example.com"})
    return response.json()
```

### テスト命名規則

```python
# ❌ Bad: 何をテストしているかわからない
def test_user():
    ...

def test_1():
    ...

# ✅ Good: テスト対象_条件_期待結果
def test_create_user_duplicate_email_returns_409():
    ...

def test_get_user_not_found_returns_404():
    ...

def test_patch_user_name_only_keeps_email():
    ...
```

### conftest.py でテストの共通化

**同じセットアップコードを各テストファイルで繰り返さない**:

```python
# ❌ Bad: 毎回同じセットアップ
def test_something():
    client = TestClient(app)
    client.post("/users", json={"name": "Alice", "email": "alice@example.com"})
    ...

def test_another():
    client = TestClient(app)
    client.post("/users", json={"name": "Alice", "email": "alice@example.com"})
    ...

# ✅ Good: フィクスチャで共通化
@pytest.fixture
def client():
    return TestClient(app)

@pytest.fixture
def alice(client):
    response = client.post("/users", json={"name": "Alice", "email": "alice@example.com"})
    return response.json()

def test_something(client, alice):
    ...

def test_another(client, alice):
    ...
```

## Practice（練習課題）

### 課題: 以下のテストを整理せよ

```python
# test_all.py に全テストが1ファイルに入っている
def test_create_user():
    ...

def test_create_user_duplicate_email():
    ...

def test_get_user_not_found():
    ...

def test_list_users_empty():
    ...

def test_patch_user_partial():
    ...

def test_name_validation():
    ...

def test_email_format():
    ...
```

**Q1.** このテストファイルの問題点は何か？
**Q2.** unit / integration に分類して、適切なファイル構成を提案せよ。
**Q3.** `conftest.py` に切り出すべき共通処理は何か？

### 模範解答

<details>
<summary>クリックで表示</summary>

**A1.** 全テストが1ファイルにまとまっていて、テストの種類（unit/integration）が混在している。ファイルが大きくなると見通しが悪くなり、特定のテストだけ実行しにくい。

**A2.**
```
tests/
├── conftest.py
├── unit/
│   └── test_validators.py       # test_name_validation, test_email_format
└── integration/
    └── test_users_api.py        # test_create_user, test_create_user_duplicate_email,
                                 # test_get_user_not_found, test_list_users_empty,
                                 # test_patch_user_partial
```

**A3.**
- `TestClient` の生成
- テスト用ユーザーの作成フィクスチャ
- テスト用 DB のセットアップ/クリーンアップ

</details>

## Mission（本番課題）

### 状況

TL からの依頼:

> 「P02-01 〜 P02-05 で色々バグ修正してテスト追加してくれたけど、テストが散らばっていてどこに何があるかわかりにくい。unit と integration に分けて、conftest.py で共通化してまとめ直してほしい。ついでにカバレッジも確認して。」

### やること

1. **現状のテストを棚卸しする**
   - 既存テストファイルを全て確認
   - 各テストが unit / integration のどちらか分類する
   ```bash
   pytest tests/ -v --co  # テスト一覧を確認（実行はしない）
   ```

2. **GitHub Issue を作成する**
   - タイトル: `refactor: テストを unit/integration に整理`
   - 本文: 現状の問題点、整理方針

3. **ブランチを作成する**
   ```bash
   git checkout -b refactor/proj02-test-organization
   ```

4. **テストディレクトリを整理する**
   ```
   tests/
   ├── conftest.py              # 共通フィクスチャ
   ├── unit/
   │   └── test_validators.py   # バリデーション系テスト
   └── integration/
       ├── test_users_api.py    # ユーザー API テスト
       └── test_teams_api.py    # チーム API テスト
   ```

5. **conftest.py を作成する**
   - `TestClient` の生成
   - テスト用ユーザー/チームの作成フィクスチャ
   - DB クリーンアップ処理

6. **全テストが通ることを確認する**
   ```bash
   pytest tests/ -v
   ```

7. **カバレッジを確認する**
   ```bash
   pytest tests/ --cov=. --cov-report=term-missing
   ```

8. **PR を作成する**
   - 変更前後のテスト構成を説明
   - カバレッジレポートを含める

### ヒント

<details>
<summary>conftest.py の書き方がわからない場合</summary>

```python
# tests/conftest.py
import pytest
from fastapi.testclient import TestClient
from main import app, Base, engine

@pytest.fixture(autouse=True)
def reset_db():
    """各テスト前に DB をリセット"""
    Base.metadata.drop_all(bind=engine)
    Base.metadata.create_all(bind=engine)
    yield

@pytest.fixture
def client():
    return TestClient(app)
```

</details>

<details>
<summary>テストの移動で import エラーが出る場合</summary>

`tests/unit/__init__.py` と `tests/integration/__init__.py` を作成してパッケージにする:

```bash
touch tests/unit/__init__.py
touch tests/integration/__init__.py
```

</details>

## Review（自己採点チェックリスト）

### 必須項目

- [ ] テストが `unit/` と `integration/` に分離されている
- [ ] `conftest.py` で共通フィクスチャが定義されている
- [ ] 全テストが通る（`pytest tests/ -v` が全て PASSED）
- [ ] P02-01〜05 の回帰テストが全て含まれている
- [ ] GitHub Issue と PR が作成されている

### 加点項目

- [ ] テスト関数名が `test_対象_条件_期待結果` の命名規則に沿っている
- [ ] カバレッジレポートが PR に含まれている
- [ ] `autouse` フィクスチャで DB リセットが自動化されている
- [ ] `pytest.ini` または `pyproject.toml` にテスト設定がある
- [ ] テストの実行速度を意識している（unit は高速、integration は中速）

### よくある間違い

- テストを移動しただけで `conftest.py` を作らない（共通化の意味がない）
- `__init__.py` を忘れて import エラーが出る
- テスト移動時にテストが減る（移動漏れ）
- カバレッジを確認しない
