## Learn（概念理解）

### Alembic マイグレーションとは

DB スキーマの変更を **バージョン管理** する仕組み。コードの Git のように、DB構造の変更履歴を追跡する:

```
alembic/versions/
├── 001_create_users.py        # users テーブル作成
├── 002_add_unique_email.py    # email に UNIQUE 制約追加
└── 003_add_teams.py           # teams テーブル作成
```

### マイグレーションが2回実行で失敗するケース

```python
# ❌ Bad: 冪等でないマイグレーション
def upgrade():
    op.create_table("users", ...)  # 2回目: テーブルが既に存在 → エラー！

def upgrade():
    op.add_column("users", Column("age", Integer))  # 2回目: カラムが既に存在 → エラー！

def upgrade():
    op.create_unique_constraint("uq_users_email", "users", ["email"])
    # 2回目: 制約が既に存在 → エラー！
```

### 冪等なマイグレーションの書き方

**何回実行しても同じ結果になる** ように書く:

```python
from alembic import op
from sqlalchemy import inspect

def upgrade():
    # テーブルの存在チェック
    bind = op.get_bind()
    inspector = inspect(bind)

    if "users" not in inspector.get_table_names():
        op.create_table("users", ...)

    # カラムの存在チェック
    columns = [c["name"] for c in inspector.get_columns("users")]
    if "age" not in columns:
        op.add_column("users", Column("age", Integer))
```

### Alembic のよくある問題と対策

| 問題 | 原因 | 対策 |
|------|------|------|
| 2回実行でエラー | 冪等でない操作 | 存在チェックを入れる |
| downgrade 未実装 | `pass` だけの downgrade | 必ず逆操作を書く |
| 順序依存の競合 | 複数人が同時にマイグレーション作成 | ブランチマージ時に整合性チェック |
| 本番とローカルの差異 | マイグレーション適用漏れ | CI で `alembic check` を実行 |

### downgrade の重要性

**ロールバックできないマイグレーションは危険**:

```python
# ❌ Bad: downgrade が空
def downgrade():
    pass

# ✅ Good: upgrade の逆操作
def upgrade():
    op.add_column("users", Column("age", Integer))

def downgrade():
    op.drop_column("users", "age")
```

```python
# ✅ Good: UNIQUE 制約の追加/削除
def upgrade():
    op.create_unique_constraint("uq_users_email", "users", ["email"])

def downgrade():
    op.drop_constraint("uq_users_email", "users")
```

### マイグレーション検証手順

```bash
# 1. 最新まで適用
alembic upgrade head

# 2. 1つ戻す
alembic downgrade -1

# 3. もう一度適用
alembic upgrade head

# 4. 全て戻す
alembic downgrade base

# 5. 最初から適用
alembic upgrade head

# → 全ステップでエラーが出なければ OK
```

## Practice（練習課題）

### 課題: 以下のマイグレーションの問題を指摘して修正せよ

```python
def upgrade():
    op.create_table(
        "categories",
        Column("id", Integer, primary_key=True),
        Column("name", String, nullable=False),
    )
    op.add_column("products", Column("category_id", Integer))

def downgrade():
    pass
```

**Q1.** このマイグレーションを2回実行するとどうなるか？
**Q2.** downgrade の問題は何か？
**Q3.** 両方修正せよ。

### 模範解答

<details>
<summary>クリックで表示</summary>

**A1.** 2回目の実行で `categories` テーブルが既に存在するためエラーになる。`products.category_id` も既に存在するためエラーになる。

**A2.** `downgrade()` が `pass` なので、ロールバックしても何も起きない。`categories` テーブルと `category_id` カラムが残ったままになる。

**A3.**
```python
from alembic import op
from sqlalchemy import Column, Integer, String, inspect

def upgrade():
    bind = op.get_bind()
    inspector = inspect(bind)

    if "categories" not in inspector.get_table_names():
        op.create_table(
            "categories",
            Column("id", Integer, primary_key=True),
            Column("name", String, nullable=False),
        )

    columns = [c["name"] for c in inspector.get_columns("products")]
    if "category_id" not in columns:
        op.add_column("products", Column("category_id", Integer))

def downgrade():
    op.drop_column("products", "category_id")
    op.drop_table("categories")
```

</details>

## Mission（本番課題）

### 状況

ローカル開発中に自分で問題を発見しました:

> 「Alembic マイグレーションを `alembic upgrade head` したあと、もう一度 `alembic upgrade head` を実行したらエラーになった。`alembic downgrade -1` → `alembic upgrade head` もエラー。CI で回したら失敗しそう...。」

### やること

1. **問題を再現する**
   ```bash
   # 1回目: 成功
   alembic upgrade head

   # 2回目: エラー
   alembic upgrade head
   ```
   - エラーメッセージを確認（テーブル/カラム/制約が既に存在するエラー）

2. **GitHub Issue を作成する**
   - タイトル: `fix: Alembic マイグレーションが冪等でない`
   - 本文: 再現手順、エラーメッセージ、影響範囲

3. **ブランチを作成する**
   ```bash
   git checkout -b fix/proj02-idempotent-migrations
   ```

4. **マイグレーションを修正する**
   - 既存のマイグレーションファイルに存在チェックを追加
   - `upgrade` と `downgrade` の両方を確認

5. **検証手順を実行する**
   ```bash
   # 全ステップでエラーなしを確認
   alembic upgrade head
   alembic downgrade -1
   alembic upgrade head
   alembic downgrade base
   alembic upgrade head
   ```

6. **検証手順をドキュメント化する**
   - マイグレーション検証の手順書を作成
   - 将来のマイグレーション作成時のチェックリスト

7. **PR を作成する**
   - Issue を参照する
   - 検証手順の結果を PR 説明に含める

### ヒント

<details>
<summary>存在チェックの書き方がわからない場合</summary>

```python

def upgrade():
    bind = op.get_bind()

    if "users" not in inspector.get_table_names():
        op.create_table(...)

    # カラム存在チェック
    columns = [c["name"] for c in inspector.get_columns("users")]
    if "new_column" not in columns:
        op.add_column(...)

    # 制約存在チェック
    constraints = inspector.get_unique_constraints("users")
    constraint_names = [c["name"] for c in constraints]
    if "uq_users_email" not in constraint_names:
        op.create_unique_constraint(...)
```

## Review（自己採点チェックリスト）

### 必須項目

- [ ] 問題を再現できた（2回実行でエラーを確認）
- [ ] マイグレーションに存在チェックを追加した
- [ ] `downgrade` が正しく実装されている（`pass` でない）
- [ ] 検証手順（upgrade → downgrade → upgrade）が全て成功する
- [ ] 検証手順がドキュメント化されている
- [ ] GitHub Issue と PR が作成されている

### 加点項目

- [ ] 全マイグレーションファイルを確認・修正した
- [ ] CI でマイグレーション検証を自動化する提案をした
- [ ] マイグレーション作成時のチェックリストを作成した
- [ ] `alembic check` コマンドの活用を検討した

### よくある間違い

- `upgrade` だけ修正して `downgrade` を放置する
- 存在チェックを入れたが、チェック自体にバグがある（テーブル名の typo など）
- 検証を1回通しただけで、downgrade → upgrade のサイクルを確認しない
- 既存データがある状態での検証を忘れる
