## Learn（概念理解）

### CRUD とは

**Create / Read / Update / Delete** の4つの基本操作:

| 操作 | HTTP メソッド | エンドポイント例 | 説明 |
|------|-------------|---------------|------|
| Create | POST | `/projects` | 新規作成 |
| Read | GET | `/projects`, `/projects/{id}` | 一覧取得、個別取得 |
| Update | PUT/PATCH | `/projects/{id}` | 更新 |
| Delete | DELETE | `/projects/{id}` | 削除 |

### Alembic マイグレーションの流れ

```bash
# 1. モデルを定義する（main.py または models.py）
class Project(Base):
    __tablename__ = "projects"
    id = Column(Integer, primary_key=True)
    name = Column(String(255), nullable=False, unique=True)
    ...

# 2. マイグレーションを自動生成
alembic revision --autogenerate -m "add projects table"

# 3. 生成されたマイグレーションファイルを確認・編集
# alembic/versions/xxxx_add_projects_table.py

# 4. マイグレーションを適用
alembic upgrade head

# 5. DB を確認
psql atlas -c "\d projects"
```

### SQLAlchemy モデル定義のベストプラクティス

```python
# ❌ Bad: 制約が不十分
class Project(Base):
    __tablename__ = "projects"
    id = Column(Integer, primary_key=True)
    name = Column(String)  # 長さ未定義、重複可能

# ✅ Good: 制約を明示
class Project(Base):
    __tablename__ = "projects"
    id = Column(Integer, primary_key=True, index=True)
    name = Column(String(255), nullable=False, unique=True)
    description = Column(Text, nullable=True)
    created_at = Column(DateTime, nullable=False, server_default=func.now())

    def __repr__(self):
        return f"<Project(id={self.id}, name={self.name})>"
```

### Pydantic スキーマ: リクエスト vs レスポンス

```python
# リクエスト用（作成時）
class ProjectCreate(BaseModel):
    name: str
    description: str | None = None

# レスポンス用（DB から取得）
class ProjectResponse(BaseModel):
    id: int
    name: str
    description: str | None
    created_at: datetime

    class Config:
        from_attributes = True  # SQLAlchemy モデルから変換可能にする
```

### REST API の設計: ステータスコードとレスポンス

| 操作 | 成功ステータス | レスポンス |
|------|-------------|----------|
| POST `/projects` | 201 Created | 作成されたリソース |
| GET `/projects` | 200 OK | リソースの配列 |
| GET `/projects/{id}` | 200 OK | 単一リソース |
| PUT `/projects/{id}` | 200 OK | 更新後のリソース |
| DELETE `/projects/{id}` | 204 No Content | なし（または 200） |

```python
# ❌ Bad: POST で 200 を返す
@app.post("/projects")
def create_project(...):
    # ...
    return project  # デフォルトで 200

# ✅ Good: POST で 201 を返す
@app.post("/projects", status_code=201)
def create_project(...):
    # ...
    return project
```

## Practice（練習課題）

### 課題: 以下のコードの問題点を見つけて修正せよ

```python
# モデル定義
class Category(Base):
    __tablename__ = "categories"
    id = Column(Integer, primary_key=True)
    name = Column(String)

# エンドポイント
@app.post("/categories")
def create_category(category: dict, db: Session = Depends(get_db)):
    db_category = Category(name=category["name"])
    db.add(db_category)
    db.commit()
    return db_category
```

**Q1.** モデル定義の問題点を3つ挙げよ。
**Q2.** エンドポイントの問題点を3つ挙げよ。
**Q3.** 修正したコードを書け。

### 模範解答

<details>
<summary>クリックで表示</summary>

**A1. モデル定義の問題点**:
1. `String` に長さ制限がない → `String(255)` など指定すべき
2. `name` に `unique=True` がない → 重複カテゴリが作成できる
3. `created_at` カラムがない → 作成日時を記録できない

**A2. エンドポイントの問題点**:
1. `category: dict` ではなく Pydantic スキーマを使うべき（バリデーション不足）
2. `status_code=201` を指定していない → POST は 201 を返すべき
3. `db.refresh(db_category)` がない → 作成後の `id` や `created_at` が含まれない可能性

**A3. 修正コード**:

```python
from pydantic import BaseModel
from sqlalchemy import func

# モデル定義
class Category(Base):
    __tablename__ = "categories"
    id = Column(Integer, primary_key=True, index=True)
    name = Column(String(255), nullable=False, unique=True)
    created_at = Column(DateTime, nullable=False, server_default=func.now())

# Pydantic スキーマ
class CategoryCreate(BaseModel):
    name: str

class CategoryResponse(BaseModel):
    id: int
    name: str
    created_at: datetime

    class Config:
        from_attributes = True

# エンドポイント
@app.post("/categories", status_code=201, response_model=CategoryResponse)
def create_category(category: CategoryCreate, db: Session = Depends(get_db)):
    db_category = Category(name=category.name)
    db.add(db_category)
    db.commit()
    db.refresh(db_category)
    return db_category
```

</details>

### ヒント

<details>
<summary>DELETE で 204 を返す方法がわからない場合</summary>

```python
from fastapi import status

@app.delete("/projects/{project_id}", status_code=status.HTTP_204_NO_CONTENT)
def delete_project(project_id: int, db: Session = Depends(get_db)):
    project = db.query(Project).filter(Project.id == project_id).first()
    if not project:
        raise HTTPException(status_code=404, detail="Project not found")
    db.delete(project)
    db.commit()
    return None  # 204 は body を返さない
```

</details>

<details>
<summary>一覧取得でソート順を指定したい場合</summary>

```python
@app.get("/projects")
def list_projects(db: Session = Depends(get_db)):
    projects = db.query(Project).order_by(Project.created_at.desc()).all()
    return projects
```

</details>

- [ ] 全テストが通る（`pytest -v`）
- [ ] PR が作成され、Issue を参照している

### 加点項目

- [ ] Pydantic スキーマで `response_model` を指定している
- [ ] 重複 name で 409 を返すテストがある
- [ ] 一覧取得が `created_at` の降順でソートされている
- [ ] DELETE で削除後の GET が 404 を返すことをテストしている
- [ ] マイグレーションファイルに downgrade が定義されている

### よくある間違い

- POST で 200 を返している（201 を返すべき）
- DELETE で 200 を返している（204 が適切）
- name の UNIQUE 制約がない（重複プロジェクトが作成できる）
- 404 のテストがない（異常系のテストは必須）
- `db.refresh()` を忘れて、作成直後の `id` や `created_at` が返らない
- マイグレーションを適用せずにテストを実行してエラーになる
