## Learn（概念理解）

### E2E（End-to-End）テストとは

**実際のユーザーの操作フローを再現する**テスト:

| テストの種類 | 対象 | 例 |
|------------|------|-----|
| **Unit テスト** | 関数単体 | `validate_status_transition()` のテスト |
| **Integration テスト** | 単一エンドポイント | `POST /tasks` のテスト |
| **E2E テスト** | 複数エンドポイントの組み合わせ | プロジェクト作成 → タスク作成 → 更新 → 削除 |

### E2E テストの設計

**実際のユースケースをシナリオ化する**:

```
シナリオ: プロジェクトマネージャーがタスクを管理する

1. プロジェクトを作成する
2. プロジェクトにタスクを3つ追加する
3. タスクのステータスを TODO → IN_PROGRESS → DONE と更新する
4. フィルタリングで IN_PROGRESS のタスクだけを取得する
5. プロジェクトサマリーで進捗を確認する
6. タスクの変更履歴を確認する
7. 完了したタスクを削除する
8. プロジェクトを削除する（cascade delete でタスクも削除される）
```

### E2E テストの構造

```python
def test_project_task_full_workflow(client, db):
    """プロジェクトとタスクの全フロー E2E テスト"""

    # 1. プロジェクト作成
    project_response = client.post("/projects", json={"name": "My Project"})
    assert project_response.status_code == 201
    project_id = project_response.json()["id"]

    # 2. タスク作成
    task1 = client.post("/tasks", json={
        "project_id": project_id,
        "title": "Task 1",
        "status": "TODO",
        "priority": "HIGH"
    })
    assert task1.status_code == 201
    task1_id = task1.json()["id"]

    # 3. ステータス更新
    update_response = client.put(f"/tasks/{task1_id}", json={"status": "IN_PROGRESS"})
    assert update_response.status_code == 200
    assert update_response.json()["status"] == "IN_PROGRESS"

    # 4. 変更履歴確認
    activities_response = client.get(f"/tasks/{task1_id}/activities")
    assert activities_response.status_code == 200
    activities = activities_response.json()
    assert len(activities) == 1
    assert activities[0]["field_name"] == "status"

    # 5. プロジェクトサマリー
    summary_response = client.get(f"/projects/{project_id}/summary")
    assert summary_response.status_code == 200
    summary = summary_response.json()
    assert summary["total_tasks"] == 1
    assert summary["tasks_by_status"]["IN_PROGRESS"] == 1

    # 6. タスク削除
    delete_response = client.delete(f"/tasks/{task1_id}")
    assert delete_response.status_code == 204

    # 7. プロジェクト削除
    delete_project_response = client.delete(f"/projects/{project_id}")
    assert delete_project_response.status_code == 204
```

### テストシナリオの設計原則

**GIVEN-WHEN-THEN** パターン:

```python
def test_filter_tasks_by_status_and_priority():
    """
    GIVEN: プロジェクトに複数のタスクがある
    WHEN: status=TODO & priority=HIGH でフィルタする
    THEN: 該当するタスクだけが返される
    """
    # GIVEN
    project = create_test_project(client)
    client.post("/tasks", json={"project_id": project["id"], "title": "Task 1", "status": "TODO", "priority": "HIGH"})
    client.post("/tasks", json={"project_id": project["id"], "title": "Task 2", "status": "TODO", "priority": "LOW"})
    client.post("/tasks", json={"project_id": project["id"], "title": "Task 3", "status": "DONE", "priority": "HIGH"})

    # WHEN
    response = client.get("/tasks?status=TODO&priority=HIGH")

    # THEN
    assert response.status_code == 200
    data = response.json()
    assert data["total"] == 1
    assert data["items"][0]["title"] == "Task 1"
```

### フィクスチャの活用


### テストの独立性

各テストは **独立して実行可能** であるべき:

```python
# ❌ Bad: テスト間で DB 状態を共有
def test_create_task():
    # タスク作成
    ...

def test_list_tasks():
    # 前のテストで作ったタスクが残っている前提
    response = client.get("/tasks")
    assert len(response.json()) == 1  # 前のテストに依存

# ✅ Good: 各テストで DB をリセット
@pytest.fixture(autouse=True)
def reset_db():
    Base.metadata.drop_all(bind=engine)
    Base.metadata.create_all(bind=engine)
    yield

def test_list_tasks():
    # このテスト内でタスクを作成
    project = create_test_project(client)
    client.post("/tasks", json={"project_id": project["id"], "title": "Task 1"})

    response = client.get("/tasks")
    assert len(response.json()["items"]) == 1
```

```python
def test_scenario_1_project_task_basic_workflow(client):
    """シナリオ1: プロジェクトとタスクの基本フロー"""

    # プロジェクト作成
    project_response = client.post("/projects", json={"name": "E2E Project"})
    assert project_response.status_code == 201
    project_id = project_response.json()["id"]

    # タスク3つ作成
    task1 = client.post("/tasks", json={
        "project_id": project_id,
        "title": "Task 1",
        "status": "TODO",
        "priority": "HIGH"
    }).json()

    task2 = client.post("/tasks", json={
        "project_id": project_id,
        "title": "Task 2",
        "status": "IN_PROGRESS",
        "priority": "MEDIUM"
    }).json()

    task3 = client.post("/tasks", json={
        "project_id": project_id,
        "title": "Task 3",
        "status": "DONE",
        "priority": "LOW"
    }).json()

    # Task 1 を TODO → IN_PROGRESS → DONE
    client.put(f"/tasks/{task1['id']}", json={"status": "IN_PROGRESS"})
    final_response = client.put(f"/tasks/{task1['id']}", json={"status": "DONE"})
    assert final_response.json()["status"] == "DONE"

    # status=IN_PROGRESS でフィルタ
    filter_response = client.get("/tasks?status=IN_PROGRESS")
    assert filter_response.status_code == 200
    filtered_tasks = filter_response.json()["items"]
    assert len(filtered_tasks) == 1
    assert filtered_tasks[0]["id"] == task2["id"]

    # プロジェクトサマリー
    summary_response = client.get(f"/projects/{project_id}/summary")
    assert summary_response.status_code == 200
    summary = summary_response.json()
    assert summary["total_tasks"] == 3
    assert summary["tasks_by_status"]["DONE"] == 2  # task1 と task3
    assert summary["tasks_by_status"]["IN_PROGRESS"] == 1  # task2

    # Task 3 を削除
    delete_response = client.delete(f"/tasks/{task3['id']}")
    assert delete_response.status_code == 204
```

```python
def test_scenario_2_status_transition_validation(client):
    """シナリオ2: ステータス遷移バリデーション"""

    project = client.post("/projects", json={"name": "Transition Test"}).json()

    }).json()

    # TODO → IN_PROGRESS（成功）

    response2 = client.put(f"/tasks/{task['id']}", json={"status": "DONE"})
    assert response2.status_code == 200

    # DONE → TODO（失敗: 422）
    response3 = client.put(f"/tasks/{task['id']}", json={"status": "TODO"})
    assert response3.status_code == 422
    assert "transition" in response3.json()["detail"].lower()
```

## Review（自己採点チェックリスト）

### 必須項目

- [ ] シナリオ2（ステータス遷移）が実装されている

- [ ] 各シナリオが独立して実行可能
- [ ] 全テストが通る（`pytest -v`）

### 加点項目

- [ ] 各テストに docstring で説明がある
- [ ] フィクスチャで共通のセットアップを再利用している
- [ ] テストが GIVEN-WHEN-THEN の構造になっている
- [ ] エッジケース（空のプロジェクト、存在しない ID など）もテストしている

### よくある間違い

- テスト間で DB 状態を共有する（独立性がない）
- アサートが不十分（ステータスコードだけで、レスポンス内容を確認しない）
- エラーケースをテストしない（422 のテストがない）
- テスト名が不明瞭（`test_1`, `test_2` など）
- フィクスチャを使わず、各テストで同じセットアップコードを繰り返す
- E2E テストなのに1つのエンドポイントしかテストしていない
