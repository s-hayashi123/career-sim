## Learn（概念理解）

### 外部キー制約違反のハンドリング

外部キー制約があると、**存在しない ID を参照した場合にエラー**が発生する:

```python
# ❌ Bad: IntegrityError を無視 → 500 Internal Server Error
@app.post("/tasks")
def create_task(task: TaskCreate, db: Session = Depends(get_db)):
    db_task = Task(**task.dict())
    db.add(db_task)
    db.commit()  # 存在しない project_id → IntegrityError → 500!
    return db_task

# ✅ Good: 事前にバリデーション → 422 Unprocessable Entity
@app.post("/tasks")
def create_task(task: TaskCreate, db: Session = Depends(get_db)):
    # 外部キーの存在確認
    project = db.query(Project).filter(Project.id == task.project_id).first()
    if not project:
        raise HTTPException(
            status_code=422,
            detail=f"Project with id {task.project_id} does not exist"
        )

    db_task = Task(**task.dict())
    db.add(db_task)
    db.commit()
    db.refresh(db_task)
    return db_task
```

### HTTP ステータスコードの使い分け

| コード | 意味 | 使い所 |
|-------|------|--------|
| 400 | Bad Request | リクエスト形式が不正（JSON が壊れているなど） |
| 404 | Not Found | リソースが存在しない（`GET /users/999` など） |
| 409 | Conflict | リソースが既存のものと競合（重複作成など） |
| 422 | Unprocessable Entity | **入力値が不正**（存在しない ID、範囲外の値など） |
| 500 | Internal Server Error | サーバー側のバグ（予期しないエラー） |

**422 の典型的な用途**:
- 存在しない外部キーを参照した
- 日付の範囲が不正（start_date > end_date）
- 列挙型（Enum）にない値を指定した

### IntegrityError のキャッチ vs 事前バリデーション

| 方法 | メリット | デメリット |
|------|---------|----------|
| **事前バリデーション** | エラーメッセージが明確<br>422 を返せる | クエリが1回増える |
| **IntegrityError キャッチ** | クエリが1回で済む | エラーメッセージが DB 依存<br>409 になる |

**ベストプラクティス**:
- **外部キー**: 事前バリデーション（422）
- **UNIQUE 制約**: IntegrityError をキャッチ（409）

```python
# 外部キー → 事前バリデーション
project = db.query(Project).filter(Project.id == task.project_id).first()
if not project:
    raise HTTPException(status_code=422, detail="Project does not exist")

# UNIQUE 制約 → IntegrityError をキャッチ
try:
    db.commit()
except IntegrityError:
    db.rollback()
    raise HTTPException(status_code=409, detail="Email already exists")
```

### エラーメッセージの設計

```python
# ❌ Bad: エラーメッセージが不親切
raise HTTPException(status_code=422, detail="Invalid input")

# ✅ Good: 何が問題かを明確に
raise HTTPException(
    status_code=422,
    detail=f"Project with id {task.project_id} does not exist"
)

# ✅ Better: フィールド名も含める
raise HTTPException(
    status_code=422,
    detail={
        "field": "project_id",
        "value": task.project_id,
        "message": "Project does not exist"
    }
)
```

def create_comment(comment: CommentCreate, db: Session = Depends(get_db)):

## Mission（本番課題）

### 状況

QA チームから以下のバグ報告が来ました:

> 「POST /tasks で存在しない project_id を指定すると、500 Internal Server Error が返ります。再現手順:
> ```bash
> curl -X POST http://localhost:8000/tasks \
>   -H "Content-Type: application/json" \
>   -d '{"project_id": 9999, "title": "Test Task"}'
> ```
> → 500 エラー（ログに IntegrityError: FOREIGN KEY constraint failed）
>
> 期待する動作: 422 Unprocessable Entity + エラーメッセージ」

### やること

1. **バグを再現する**
   - 存在しない project_id で POST /tasks を叩く
   - 500 エラーとログの IntegrityError を確認

2. **GitHub Issue を作成する**
   - タイトル: `fix: POST /tasks が存在しない project_id で 500 を返す`
   - 本文: 再現手順・期待する動作（422）・実際の動作（500）

   <details>

   ```bash

<summary>project_id の存在確認コード</summary>

```python
@app.post("/tasks", status_code=201)
def create_task(task: TaskCreate, db: Session = Depends(get_db)):
    # project_id の存在確認
    project = db.query(Project).filter(Project.id == task.project_id).first()
    if not project:
        raise HTTPException(
            status_code=422,
            detail=f"Project with id {task.project_id} does not exist"
        )

    # assignee_id の存在確認（指定された場合のみ）
    if task.assignee_id is not None:
        assignee = db.query(User).filter(User.id == task.assignee_id).first()
        if not assignee:
            raise HTTPException(
                status_code=422,
                detail=f"User with id {task.assignee_id} does not exist"
            )

    db_task = Task(**task.dict())
    db.add(db_task)
    db.commit()
    db.refresh(db_task)
    return db_task
```
