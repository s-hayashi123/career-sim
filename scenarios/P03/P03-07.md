## Learn（概念理解）

### 状態遷移図（State Transition Diagram）

ビジネスロジックとして、**許可された遷移のみを受け入れる**:

```
TODO → IN_PROGRESS → DONE

許可される遷移:
- TODO → IN_PROGRESS ✅
- IN_PROGRESS → DONE ✅
- TODO → DONE ✅（直接完了も許可する場合）

許可されない遷移:
- DONE → TODO ❌
- DONE → IN_PROGRESS ❌
- IN_PROGRESS → TODO ❌（巻き戻し不可）
```

### 状態遷移バリデーションの実装

```python
# ❌ Bad: 遷移ルールがない
@app.put("/tasks/{task_id}")
def update_task(task_id: int, task: TaskUpdate, db: Session = Depends(get_db)):
    db_task = db.query(Task).filter(Task.id == task_id).first()
    if not db_task:
        raise HTTPException(status_code=404, detail="Task not found")

    # 任意のステータスに変更可能（DONE → TODO も可能）
    if task.status:
        db_task.status = task.status

    db.commit()
    return db_task

# ✅ Good: 遷移ルールを定義
ALLOWED_STATUS_TRANSITIONS = {
    TaskStatus.TODO: [TaskStatus.IN_PROGRESS, TaskStatus.DONE],
    TaskStatus.IN_PROGRESS: [TaskStatus.DONE],
    TaskStatus.DONE: []  # DONE からは遷移不可
}

def validate_status_transition(current: TaskStatus, new: TaskStatus) -> bool:
    """ステータス遷移が許可されているか検証"""
    if current == new:
        return True  # 同じステータスへの変更は許可
    return new in ALLOWED_STATUS_TRANSITIONS.get(current, [])

@app.put("/tasks/{task_id}")
def update_task(task_id: int, task: TaskUpdate, db: Session = Depends(get_db)):
    db_task = db.query(Task).filter(Task.id == task_id).first()
    if not db_task:
        raise HTTPException(status_code=404, detail="Task not found")

    # ステータス変更の場合、遷移ルールを検証
    if task.status and task.status != db_task.status:
        if not validate_status_transition(db_task.status, task.status):
            raise HTTPException(
                status_code=422,
                detail=f"Invalid status transition: {db_task.status.value} → {task.status.value}"
            )
        db_task.status = task.status

    # 他のフィールドも更新...
    db.commit()
    db.refresh(db_task)
    return db_task
```

### ビジネスロジックの設計判断

**どの遷移を許可するか**はビジネス要件次第:

| パターン | 遷移ルール | 用途 |
|---------|----------|------|
| **一方向のみ** | TODO → IN_PROGRESS → DONE<br>（巻き戻し不可） | タスク管理、注文処理 |
| **巻き戻し許可** | TODO ⇄ IN_PROGRESS ⇄ DONE | 開発タスク（再オープン可） |
| **完了後の再オープン** | DONE → TODO のみ許可 | サポートチケット |

**ADR で設計判断を記録する**:
- なぜこの遷移ルールにしたか
- 他の選択肢（巻き戻し許可など）を検討したか
- 将来の変更可能性

### exclude_unset の活用

PATCH（部分更新）では、**指定されたフィールドだけを更新する**:

```python
# TaskUpdate スキーマ（全て任意）
class TaskUpdate(BaseModel):
    title: str | None = None
    description: str | None = None
    status: TaskStatus | None = None
    priority: TaskPriority | None = None
    assignee_id: int | None = None

# エンドポイント
@app.put("/tasks/{task_id}")
def update_task(task_id: int, task: TaskUpdate, db: Session = Depends(get_db)):
    db_task = db.query(Task).filter(Task.id == task_id).first()
    if not db_task:
        raise HTTPException(status_code=404, detail="Task not found")

    # exclude_unset: 指定されたフィールドだけを取得
    update_data = task.dict(exclude_unset=True)

    # status の遷移検証
    if "status" in update_data:
        new_status = update_data["status"]
        if new_status != db_task.status:
            if not validate_status_transition(db_task.status, new_status):
                raise HTTPException(status_code=422, detail="Invalid status transition")

    # フィールドを更新
    for field, value in update_data.items():
        setattr(db_task, field, value)

    db.commit()
    db.refresh(db_task)
    return db_task
```

## Practice（練習課題）

### 課題: 注文ステータスの遷移ルールを実装せよ

以下の注文ステータスがあるとする:

```python
class OrderStatus(str, Enum):
    PENDING = "PENDING"      # 保留中
    CONFIRMED = "CONFIRMED"  # 確定
    SHIPPED = "SHIPPED"      # 発送済み
    DELIVERED = "DELIVERED"  # 配達済み
    CANCELLED = "CANCELLED"  # キャンセル
```

**ビジネスルール**:
- PENDING → CONFIRMED または CANCELLED
- CONFIRMED → SHIPPED または CANCELLED
- SHIPPED → DELIVERED（キャンセル不可）
- DELIVERED / CANCELLED は最終状態（遷移不可）

**Q1.** `ALLOWED_TRANSITIONS` 辞書を定義せよ。
**Q2.** SHIPPED → CANCELLED への遷移を試みた場合、どうなるべきか？
**Q3.** 遷移検証関数を実装せよ。

```python
ALLOWED_ORDER_TRANSITIONS = {
```

## Mission（本番課題）

### 状況

TL からコードレビューのフィードバックが来ました:

> 「タスクのステータス遷移が自由すぎます。DONE から TODO に戻せてしまうのは問題。以下の遷移ルールを実装してください:
> - TODO → IN_PROGRESS または DONE
> - IN_PROGRESS → DONE
> - DONE → 遷移不可（最終状態）
>
> 遷移ルールは ADR-005 として記録してください。」

   - タイトル: `feat: タスクステータスの遷移バリデーションを追加`
   - 本文: 現状の問題点（自由に遷移可能）、要件（遷移ルール）

   ```bash
   git checkout -b feat/proj03-status-transition-validation
   ```

3. **ADR-005 を作成する**

   - Context: 現状の問題点、ビジネス要件

   - Consequences: メリット（データ整合性）、デメリット（柔軟性低下）

   ```python
       TaskStatus.TODO: [TaskStatus.IN_PROGRESS, TaskStatus.DONE],

       TaskStatus.DONE: []

   ```

   - 許可される遷移（TODO → IN_PROGRESS, IN_PROGRESS → DONE）

### ヒント

<details>
<summary>ADR-005 のテンプレート</summary>

```markdown
# ADR-005: タスクステータス遷移ルール

## Status
Accepted

## Context
現状、タスクのステータスは任意の値に変更可能。これにより以下の問題が発生:
- 完了（DONE）したタスクを TODO に巻き戻せる → 進捗管理が不正確
- データの整合性が保証されない

ビジネス要件として、タスクは以下のライフサイクルを持つ:
1. 作成時は TODO
2. 作業開始で IN_PROGRESS
3. 完了で DONE（最終状態）

## Decision

### 許可される遷移
- TODO → IN_PROGRESS
- TODO → DONE（直接完了も許可）
- IN_PROGRESS → DONE

### 許可されない遷移
- DONE → TODO（巻き戻し不可）
- DONE → IN_PROGRESS（巻き戻し不可）
- IN_PROGRESS → TODO（巻き戻し不可）

### 実装
- `ALLOWED_STATUS_TRANSITIONS` 辞書で遷移ルールを定義
- PUT /tasks/{id} で遷移ルールを検証
- 不正な遷移は 422 Unprocessable Entity

## Consequences

**Pros**:
- データの整合性が保証される
- ビジネスルールが明確になる
- 不正な操作を防げる

**Cons**:
- 完了したタスクを再オープンできない（仕様変更が必要な場合は ADR を更新）
- 柔軟性が低下する

**将来の拡張性**:
- 再オープン機能が必要な場合、DONE → TODO を許可する選択肢もある
- その場合は ADR を更新し、再オープン時のログを記録する仕組みを追加
```

</details>

```python
def test_valid_transition_todo_to_in_progress(client):
    # タスク作成（デフォルトで TODO）
    project = create_test_project(client)
    task_response = client.post("/tasks", json={
        "project_id": project["id"],
        "title": "Task 1",
        "status": "TODO"
    })
    task_id = task_response.json()["id"]

    # TODO → IN_PROGRESS（許可）
    response = client.put(f"/tasks/{task_id}", json={"status": "IN_PROGRESS"})
    assert response.status_code == 200
    assert response.json()["status"] == "IN_PROGRESS"

def test_invalid_transition_done_to_todo(client):
    # タスクを DONE まで遷移
    project = create_test_project(client)
    task_response = client.post("/tasks", json={
        "project_id": project["id"],
        "title": "Task 1",
        "status": "DONE"
    })
    task_id = task_response.json()["id"]

    # DONE → TODO（不許可）
    response = client.put(f"/tasks/{task_id}", json={"status": "TODO"})
    assert response.status_code == 422
    assert "transition" in response.json()["detail"].lower()

def test_same_status_allowed(client):
    # 同じステータスへの変更は許可
    project = create_test_project(client)
    task_response = client.post("/tasks", json={
        "project_id": project["id"],
        "title": "Task 1",
        "status": "TODO"
    })
    task_id = task_response.json()["id"]

    response = client.put(f"/tasks/{task_id}", json={"status": "TODO"})
    assert response.status_code == 200
```

### 加点項目

- [ ] ADR に将来の拡張性（再オープン機能）について言及している

### よくある間違い

- ADR を書かずに実装だけする（設計判断が記録されない）
- 遷移検証を POST /tasks にも追加する（作成時は検証不要）
- 同じステータスへの変更を拒否する（冪等性が失われる）
- エラーメッセージが不親切（「Invalid transition」だけ）
- exclude_unset を使わず、未指定のフィールドが None になる
- 遷移ルールをハードコードせず、辞書で管理しない（拡張性が低い）
