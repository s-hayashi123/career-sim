# P03-01: PM要望からデータモデル設計・ADR作成

> **シナリオ**: PMから「プロジェクト管理・タスク管理機能が欲しい」
> **想定時間**: 5h
> **スキルタグ**: P1（コードリーディング）, P2（DB設計）, P8（ドキュメント・プロセス改善）

---

## Learn（概念理解）

### なぜ要件定義とデータモデル設計が重要か

実務では、いきなりコードを書き始めるのではなく **要件を整理してデータモデルを設計する** ステップが必須:

```
要件定義 → データモデル設計 → ADR作成 → 実装 → テスト
↑ これが抜けると、後で大幅な手戻りが発生する
```

**よくある失敗パターン**:
- 要件を確認せずに実装 → PM「そうじゃないんだよね」→ 作り直し
- テーブル設計をせずに実装 → 外部キーが矛盾、後でマイグレーションが複雑化
- 設計判断を記録しない → 「なんでこうしたんだっけ？」→ 同じ議論を繰り返す

### データモデル設計の手順

1. **要件からエンティティを抽出する**
   - 「プロジェクト管理」→ Project エンティティ
   - 「タスク管理」→ Task エンティティ

2. **エンティティの属性を洗い出す**
   - Project: name, description, created_at
   - Task: title, description, status, assignee_id, priority

3. **リレーション（関連）を定義する**
   - 1つの Project に複数の Task が紐づく → 1対多
   - Task は1つの Project に属する → 多対1

4. **制約を決める**
   - UNIQUE: name は重複不可？
   - NOT NULL: description は必須？
   - DEFAULT: status のデフォルト値は？

### ER図（Entity Relationship Diagram）

視覚的にテーブル構造を表現する図:

```
┌─────────────┐       ┌─────────────┐
│  projects   │       │   tasks     │
├─────────────┤       ├─────────────┤
│ id (PK)     │<──┐   │ id (PK)     │
│ name        │   └───│ project_id  │ (FK)
│ description │       │ title       │
│ created_at  │       │ description │
└─────────────┘       │ status      │
                      │ assignee_id │
                      │ priority    │
                      │ created_at  │
                      └─────────────┘

 (1) ────────── (多)
```

### ADR（Architecture Decision Record）とは

**設計判断を記録するドキュメント**。後から「なぜそうしたか」を振り返れる:

```markdown
# ADR-004: プロジェクト管理のデータモデル設計

## Status
Accepted

## Context
PMから「プロジェクトとタスクを管理する機能」の要望。
- 1プロジェクトに複数タスクが紐づく
- タスクはステータス（TODO/IN_PROGRESS/DONE）を持つ
- タスクは担当者と優先度を持つ

## Decision
以下のテーブル構造を採用:
- projects テーブル: id, name, description, created_at
- tasks テーブル: id, project_id (FK), title, description, status, assignee_id, priority, created_at

## Consequences
- 1対多のリレーションで柔軟にタスク追加可能
- ステータス遷移バリデーションは後のシナリオで実装
- assignee_id は users テーブルへの外部キーとする
```

### 正規化の基本

**データの重複を排除する** 設計手法:

```python
# ❌ Bad: 非正規化（重複データ）
tasks:
  - id: 1
    title: "タスク1"
    project_name: "Project A"  # プロジェクト名を task に持つ
  - id: 2
    title: "タスク2"
    project_name: "Project A"  # 同じ名前が重複

# Project A の名前変更 → 全タスクを更新する必要がある（更新異常）

# ✅ Good: 正規化（外部キーで参照）
projects:
  - id: 1
    name: "Project A"

tasks:
  - id: 1
    title: "タスク1"
    project_id: 1  # 外部キー
  - id: 2
    title: "タスク2"
    project_id: 1  # 外部キー

# Project の名前変更 → projects テーブルの1レコードだけ更新すればOK
```

## Practice（練習課題）

### 課題: 以下の要件からデータモデルを設計せよ

**要件**:
- ブログシステムを作りたい
- 記事（Article）は title, content, published_at を持つ
- コメント（Comment）は記事に紐づく
- コメントは author, content を持つ
- 1つの記事に複数のコメントがつく

**Q1.** エンティティとその属性を洗い出せ。
**Q2.** リレーション（1対多、多対1）を定義せよ。
**Q3.** 簡単な ER 図（テキスト形式）を書け。

### 模範解答

<details>
<summary>クリックで表示</summary>

**A1. エンティティと属性**:

```
articles:
  - id (PK)
  - title
  - content
  - published_at

comments:
  - id (PK)
  - article_id (FK → articles.id)
  - author
  - content
  - created_at
```

**A2. リレーション**:
- 1つの Article に複数の Comment が紐づく（1対多）
- 1つの Comment は1つの Article に属する（多対1）

**A3. ER 図**:

```
┌─────────────┐       ┌─────────────┐
│  articles   │       │  comments   │
├─────────────┤       ├─────────────┤
│ id (PK)     │<──┐   │ id (PK)     │
│ title       │   └───│ article_id  │ (FK)
│ content     │       │ author      │
│ published_at│       │ content     │

                      └─────────────┘

 (1) ────────── (多)
```

## Mission（本番課題）

### 状況

あなたは `project-atlas-api` チームの開発者です。
PM から以下の要望が Slack に届きました:

> 「プロジェクト管理機能が欲しいです。要件は以下:
> - プロジェクトを作成・一覧・更新・削除できる
> - プロジェクトには name（必須）、description（任意）を持たせたい
> - プロジェクトに紐づくタスクを管理したい
> - タスクは title（必須）、description（任意）、status（TODO/IN_PROGRESS/DONE）、assignee（担当者、users テーブルへの外部キー）、priority（LOW/MEDIUM/HIGH）を持つ
> - 1つのプロジェクトに複数のタスクが紐づく
>
> まずデータモデルを設計してもらって、ADR で共有してもらえますか？」

### やること

1. **要件を整理する**
   - PM の要望から必要なエンティティと属性を抽出する
   - リレーションを整理する

2. **GitHub Issue を作成する**
   - タイトル: `feat: プロジェクト管理・タスク管理のデータモデル設計`
   - 本文: 要件の箇条書き + 受入条件（ADR-004 作成）

   <details>
   <summary>Issue の書き方例</summary>

   ```markdown
   ## 概要
   プロジェクト管理・タスク管理機能を実装するため、データモデルを設計しADRとして記録する。

   ## 背景
   PMからプロジェクト・タスク管理機能の要望があり、実装前にデータモデル設計を行う必要がある。

   ## 要件
   - [ ] プロジェクト（Project）エンティティの定義
     - name（必須、UNIQUE）
     - description（任意）
     - created_at
   - [ ] タスク（Task）エンティティの定義
     - title（必須）
     - description（任意）
     - status（TODO/IN_PROGRESS/DONE）
     - assignee_id（users への外部キー）
     - priority（LOW/MEDIUM/HIGH）
     - project_id（projects への外部キー）
     - created_at
   - [ ] 1対多のリレーション設計（Project → Tasks）
   - [ ] 外部キー制約の定義

   ## 受入条件
   - [ ] ADR-004 が `docs/adrs/` に作成されている
   - [ ] ER 図が含まれている
   - [ ] 外部キー制約（ON DELETE の挙動含む）が明記されている
   - [ ] ENUM 型（status, priority）が定義されている

   ## 参考資料
   - [ADR テンプレート](docs/adrs/ADR-001-example.md)
   ```

   **ポイント**:
   - **明確なタイトル**: 何をするかが一目でわかる
   - **背景**: なぜやるのか
   - **要件**: チェックボックスで明確化
   - **受入条件**: 完了の定義
   </details>

3. **ブランチを作成する**
   ```bash
   git checkout -b feat/proj03-data-model-design
   ```

4. **ER 図を作成する**
   - Markdown のコードブロックでテキスト形式の ER 図を書く
   - projects テーブルと tasks テーブルの関係を明確にする

5. **ADR-004 を作成する**
   - ファイル: `docs/adrs/ADR-004-project-task-data-model.md`
   - テンプレート構造:
     - Status: Accepted
     - Context: PM 要望の背景
     - Decision: テーブル設計の詳細
     - Consequences: メリット・デメリット

6. **ADR に以下を含める**
   - projects テーブルのカラム定義（型、制約）
   - tasks テーブルのカラム定義（型、制約）
   - 外部キー制約（tasks.project_id → projects.id）
   - ステータスの ENUM 型定義
   - 優先度の ENUM 型定義

7. **PR を作成する**
   - Issue を参照する
   - ADR の内容を説明する

   <details>
   <summary>PR の書き方例</summary>

   ```markdown
   ## 概要
   プロジェクト管理・タスク管理機能のデータモデル設計を ADR-004 として記録しました。

   Closes #XX

   ## 変更内容
   - ADR-004 を作成（`docs/adrs/ADR-004-project-task-data-model.md`）
   - projects テーブルのスキーマ定義
   - tasks テーブルのスキーマ定義
   - 1対多リレーション（Project → Tasks）の設計
   - ENUM 型（TaskStatus, TaskPriority）の定義

   ## データモデル概要
   ```
   projects (1) ──→ (多) tasks
   - projects: id, name, description, created_at
   - tasks: id, project_id(FK), title, description, status, assignee_id, priority, created_at
   ```

   ## 設計判断
   - **外部キー制約**: ON DELETE CASCADE（プロジェクト削除時にタスクも削除）
   - **正規化**: プロジェクト情報は projects テーブルに集約
   - **ENUM 型**: status と priority は定義された値のみ許可

   ## レビュー観点
   - [ ] テーブル設計が要件を満たしているか
   - [ ] 外部キー制約が適切か
   - [ ] ENUM 型の値が適切か
   - [ ] ADR のフォーマットが正しいか
   ```

   **ポイント**:
   - **Closes #XX**: Issue を自動クローズ
   - **変更内容**: 何をしたか明確に
   - **概要図**: 視覚的にわかりやすく
   - **設計判断**: なぜそうしたか
   - **レビュー観点**: レビュアーの負担を減らす
   </details>

### projects テーブル
| カラム | 型 | 制約 | 説明 |
|--------|-----|------|------|
| id | Integer | PK, AUTO_INCREMENT | プロジェクト ID |
| name | String(255) | NOT NULL, UNIQUE | プロジェクト名 |
| description | Text | NULL | プロジェクト説明 |
| created_at | Timestamp | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 作成日時 |

### tasks テーブル
（同様にカラム定義）

- tasks.project_id → projects.id (外部キー)

## Consequences
**Pros**:
- シンプルな1対多のリレーションで拡張性がある
- 正規化されていて、データの整合性を保てる

**Cons**:
- タスクの階層構造（サブタスク）は現状サポートしない

## Review（自己採点チェックリスト）

### 必須項目

- [ ] GitHub Issue に要件と受入条件が記載されている
- [ ] ADR-004 が `docs/adrs/` に作成されている
- [ ] ADR に projects テーブルのカラム定義がある（id, name, description, created_at）
- [ ] ADR に tasks テーブルのカラム定義がある（id, project_id, title, description, status, assignee_id, priority, created_at）
- [ ] ADR に外部キー制約（tasks.project_id → projects.id）が記載されている
- [ ] ER 図（テキスト形式でも可）が含まれている
- [ ] Status（Accepted）、Context、Decision、Consequences のセクションがある
- [ ] PR が作成され、Issue を参照している
- [ ] コミットメッセージが `docs:` プレフィックス

### 加点項目

- [ ] status と priority を ENUM 型として定義している
- [ ] ON DELETE の挙動（CASCADE / RESTRICT）を明記している
- [ ] インデックスの設計を考慮している（例: project_id, assignee_id）
- [ ] 将来の拡張性（サブタスク、タグなど）についても言及している
- [ ] NOT NULL / UNIQUE / DEFAULT 制約が適切に定義されている

### よくある間違い

- ADR を書かずにいきなり実装を始める（設計判断が記録されない）
- 外部キー制約を定義しない（データの整合性が保証されない）
- status を String 型にして、任意の文字列を許可する（ENUM で制限すべき）
- projects.name に UNIQUE 制約をつけない（重複プロジェクト名が作れてしまう）
- ON DELETE の挙動を考慮しない（プロジェクト削除時にタスクが孤立する）
- ER 図がなく、テーブル間の関係がわかりにくい
