## Learn（概念理解）

### モノリシックファイルの問題

全てのコードが1ファイルにあると:

- **読みにくい** — 200行超えると目的のコードを探すのが大変
- **変更が怖い** — 1箇所の修正が他に影響しないか不安
- **チーム開発しにくい** — 同じファイルを複数人が同時に編集 → コンフリクト多発
- **テストしにくい** — 特定の関数だけをテストするのが困難

### 一般的な FastAPI プロジェクト構造

```
app/
├── __init__.py
├── main.py          ← FastAPIアプリの初期化のみ
├── config.py        ← 設定・環境変数
├── database.py      ← DB接続設定
├── models/          ← SQLAlchemy モデル
│   ├── __init__.py
│   ├── user.py
│   └── team.py
├── schemas/         ← Pydantic スキーマ
│   ├── __init__.py
│   ├── user.py
│   └── team.py
├── routers/         ← エンドポイント定義
│   ├── __init__.py
│   ├── users.py
│   └── teams.py
└── services/        ← ビジネスロジック（後で追加）
    ├── __init__.py
    ├── user_service.py
    └── team_service.py
```

### 分離の原則: 関心の分離（Separation of Concerns）

各ファイルは **1つの責務** を持つ:

| ファイル | 責務 |
|---------|------|
| `models/` | データの形（テーブル定義） |
| `schemas/` | 入出力の形（リクエスト/レスポンス） |
| `routers/` | HTTPの処理（どのURLで何を返すか） |
| `services/` | ビジネスロジック（実際の処理） |

### ADR（Architecture Decision Record）とは

設計判断を記録するドキュメント。「なぜこう設計したか」を残す。

```markdown
# ADR-001: プロジェクト構造方針

## ステータス
提案

## コンテキスト
現在、全コード（モデル・スキーマ・ルート・DB設定）が main.py に集約されている。
コードの可読性とチーム開発効率が低い。

## 決定
以下の構造に段階的に移行する:
- models/ — SQLAlchemy モデル
- schemas/ — Pydantic スキーマ
- routers/ — エンドポイント
- services/ — ビジネスロジック（Project 04 で導入）

## 影響
- 初期コストはあるが、長期的な保守性が大幅に向上する
- テストの書きやすさも改善される
```

**ポイント**: ADR は **実装はしない** 。方針を記録するだけ。実装は Project 04（Refactor）で行う。

## Practice（練習課題）

### 課題: 以下のコードを見て、どう分割するか計画せよ

```python
# app.py（100行のモノリシック）
from fastapi import FastAPI
from sqlalchemy import ...

# DB設定（10行）
engine = create_engine(...)
...

# モデル（20行）
class Item(Base):
    ...
class Category(Base):
    ...

# スキーマ（20行）
class ItemCreate(BaseModel):
    ...
class ItemResponse(BaseModel):
    ...

# エンドポイント（50行）
@app.post("/items")
def create_item(...):
    ...
@app.get("/items")
def list_items(...):
    ...
@app.get("/categories")
def list_categories(...):
    ...
```

**Q1.** このコードを何ファイルに分割するか？各ファイル名とその中身を列挙せよ。
**Q2.** 分割の優先順位をつけるとしたら、最初にどこから手をつけるか？

### 模範解答

<details>
<summary>クリックで表示</summary>

**A1.**
| ファイル | 内容 |
|---------|------|
| `database.py` | `engine`, `SessionLocal`, `Base` |
| `models/item.py` | `Item` モデル |
| `models/category.py` | `Category` モデル |
| `schemas/item.py` | `ItemCreate`, `ItemResponse` |
| `routers/items.py` | `/items` 関連エンドポイント |
| `routers/categories.py` | `/categories` 関連エンドポイント |
| `main.py` | FastAPI アプリ初期化 + router 登録 |

**A2.** `database.py` の分離から。理由: 他の全ファイルが依存するため、ここを先に分離すると後が楽。

</details>

## Mission（本番課題）

### 状況

コードを読んでいて気付きました:

> 「`main.py` にモデル定義、スキーマ、DB設定、全エンドポイントが入ってる。これ今後機能が増えたら確実に破綻する。まずは現状を分析して、リファクタ方針を ADR に残しておこう。」

**注意: このノックでは実装（リファクタ）はしない。分析と方針策定のみ。**

### やること

1. **`main.py` を読んで分析する**
   - 何行あるか？
   - どんな責務が混在しているか？
   - 何ファイルに分割すべきか？

2. **GitHub Issue を作成する**
   - タイトル: `refactor: プロジェクト構造の改善方針を策定する`
   - 本文: 現状の問題点と改善の必要性

3. **ADR-001 を作成する**
   - ファイル: `docs/adr/ADR-001-project-structure.md`（`project-atlas-api` リポに作成）
   - 内容:
     - ステータス: 提案
     - コンテキスト: 現状の問題
     - 決定: 推奨する構造
     - 段階的移行計画（いつやるか）

4. **PR を作成する**
   - ADR ファイルのみを含む

### ADR のテンプレート

```markdown
# ADR-001: プロジェクト構造方針

## ステータス
提案

## 日付
YYYY-MM-DD

## コンテキスト
（現状の問題を具体的に記述）

## 検討した選択肢
1. 案A: ...
2. 案B: ...

## 決定
（採用した方針）

## 理由
（なぜこの方針を選んだか）

## 影響
（この決定によるメリット・デメリット・リスク）
```

## Review（自己採点チェックリスト）

### 必須項目

- [ ] `main.py` の現状を分析している（行数、混在する責務の列挙）
- [ ] ADR-001 が `docs/adr/ADR-001-project-structure.md` に作成されている

- [ ] ADR に推奨するプロジェクト構造が記載されている

- [ ] **実装（リファクタ）はしていない**（方針策定のみ）

### 加点項目

- [ ] ADR に「検討した選択肢」が複数挙げられている
- [ ] ADR に段階的移行計画（Project 04 で実施など）が記載されている
- [ ] ADR にリスクや注意点が記載されている
- [ ] 分割後のディレクトリ構造が具体的に示されている

### よくある間違い

- いきなりリファクタを始めてしまう（このノックは方針策定のみ）
- ADR が抽象的すぎる（「構造を改善する」だけでは不十分）
- 推奨する構造にサービス層を含めない（将来のことも考える）
- Issue を作らずに ADR だけ書く
