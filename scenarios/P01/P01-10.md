# P01-10: スプリントレトロスペクティブ

> **シナリオ**: Project 01 のスプリント完了、振り返りを行う
> **想定時間**: 3h
> **スキルタグ**: P8（ドキュメント・プロセス改善）

---

## Learn（概念理解）

### レトロスペクティブ（振り返り）とは

スプリント（プロジェクト期間）の終わりに **「何がうまくいったか」「何を改善すべきか」** を振り返る会議。
個人でやる場合は、自分自身の作業を振り返るドキュメントを書く。

### なぜ振り返りが重要か

- **同じミスを繰り返さない** — 失敗から学ぶ
- **良い習慣を定着させる** — うまくいったことを意識的に続ける
- **成長を可視化する** — 過去の自分と比較できる
- **ポートフォリオの証跡** — 「振り返りができる人」は実務で信頼される

### レトロスペクティブのフォーマット

```markdown
# Project 01 レトロスペクティブ

## 期間
YYYY-MM-DD 〜 YYYY-MM-DD

## 完了したノック
- P01-01: ...
- P01-02: ...
（全ノックを列挙）

## Good（うまくいったこと）
-（具体的に）

## Bad（うまくいかなかったこと）
-（具体的に）

## Try（次に試すこと）
-（具体的なアクション）

## 学んだこと
-（技術的な学び）

## 数字で振り返る
- 作成した Issue: X 件
- 作成した PR: X 件
- 作成した ADR: X 件
- テストカバレッジ: XX% → XX%
```

### スコアカードとは

各ノックの成果を **定量的に評価** する表。

```markdown
# Project 01 スコアカード

| ノック | 完了 | Issue | PR | テスト | 備考 |
|-------|------|-------|-----|--------|------|
| P01-01 | ✅ | #1 | #2 | - | README改善 |
| P01-02 | ✅ | #3 | #4 | pytest-cov導入 | カバレッジ29% |
| ...    |     |      |     |        |      |
```

### evidence-index とは

全ノックの成果物を一覧化したインデックス。ポートフォリオの根拠となる。

```markdown
| スキル | 証跡 | リンク |
|--------|------|--------|
| P3（テスト） | カバレッジ測定・改善 | PR #4 |
| P4（コード品質） | バリデーション追加 | PR #8 |
| P5（バグ修正） | 空DB時500修正 | PR #6 |
```

## Practice（練習課題）

### 課題: 以下の振り返りの改善点を指摘せよ

```markdown
## Good
- いろいろ勉強になった
- コードが書けた

## Bad
- 時間がかかった
- 難しかった

## Try
- もっと頑張る
```

### 模範解答

<details>
<summary>クリックで表示</summary>

**問題点:**
1. 全て **抽象的すぎる** 。何がうまくいって何が難しかったか不明
2. Try が「もっと頑張る」はアクションではない
3. 具体的な学びがない

**改善:**
```markdown
## Good
- Issue → ブランチ → PR の一連のGit作業フローが身についた
- Pydantic の Field() を使ったバリデーションの書き方を学んだ
- pytest-cov でカバレッジを測定する方法を覚えた

## Bad
- P01-03 のバグ修正で原因特定に1時間かかった（コードを上から順に読んでいた）
- コミットメッセージの書き方で迷って時間をロスした

## Try
- バグ修正はまずエラーメッセージを読んで該当行を特定する（上から読まない）
- Conventional Commits のチートシートをブックマークしておく
```

</details>

## Mission（本番課題）

### 状況

Project 01 の全ノック（P01-01 〜 P01-09）が完了しました。
スプリントの終わりとして、振り返りとスコアカードを作成します。

### やること

1. **レトロスペクティブを作成する**
   - ファイル: `career-sim/docs/retros/P01-retro.md`
   - フォーマットに沿って Good / Bad / Try を **具体的に** 書く
   - 学んだ技術的な内容を列挙する

2. **スコアカードを作成する**
   - ファイル: `career-sim/scorecards/P01-scorecard.md`
   - 各ノックの完了状況、Issue/PR番号、備考を記録

3. **evidence-index を更新する**
   - ファイル: `career-sim/docs/evidence-index.md`
   - Project 01 で作成した証跡（PR、ADR など）を追加

4. **自己採点する**
   - 各ノックの Review チェックリストを振り返り、達成度を記録
   - 全体を通しての達成度（%）を算出

### レトロスペクティブに含めるべき内容

- P01-01 〜 P01-09 で何をやったか（各1行サマリー）
- Good: 具体的に何がうまくいったか（最低3つ）
- Bad: 具体的に何が難しかった/時間がかかったか（最低3つ）
- Try: 次の Project 02 で具体的に試すこと（最低3つ）
- 学んだ技術: 具体的な技術名と内容
- 数字: Issue/PR/ADR の数、カバレッジの変化

## Review（自己採点チェックリスト）

### 必須項目

- [ ] レトロスペクティブ `P01-retro.md` が作成されている
- [ ] Good / Bad / Try が **具体的** に書かれている（抽象的でない）
- [ ] スコアカード `P01-scorecard.md` が作成されている
- [ ] スコアカードに全ノック（P01-01 〜 P01-09）の状況が記録されている
- [ ] `evidence-index.md` に Project 01 の証跡が追加されている

### 加点項目

- [ ] 学んだ技術が具体的に列挙されている
- [ ] 数字での振り返り（Issue/PR 数、カバレッジ変化）がある
- [ ] Bad に対する具体的な改善アクションが Try に書かれている
- [ ] 次の Project 02 への準備・心構えが書かれている

### よくある間違い

- Good/Bad/Try が抽象的（「頑張った」「難しかった」「もっと頑張る」）
- スコアカードに Issue/PR の番号を書き忘れる
- evidence-index を更新しない
- 振り返りをやらずに次のプロジェクトに進む（最も良くない）
