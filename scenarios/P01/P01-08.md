## Learn（概念理解）

### 入力バリデーションとは

ユーザーからの入力を **受け入れる前にチェック** すること。

```
❌ バリデーションなし:
  POST /users {"name": "", "email": ""} → 200 OK（空のユーザーが作られる）

✅ バリデーションあり:
  POST /users {"name": "", "email": ""} → 422 Unprocessable Entity
  {"detail": [{"msg": "名前は1文字以上必要です"}]}
```

### なぜバリデーションが必要か

1. **データの整合性** — DB に不正なデータが入ると後で問題になる
2. **セキュリティ** — 不正な入力による攻撃（SQLインジェクション等）を防ぐ
3. **UX** — 何が間違っているか明確なエラーメッセージを返す

### Pydantic でのバリデーション

```python
from pydantic import BaseModel, Field, field_validator

class UserCreate(BaseModel):
    name: str = Field(..., min_length=1, max_length=100)
    email: str = Field(...)

    # カスタムバリデータ
    @field_validator("name")
    @classmethod
    def name_must_not_be_blank(cls, v):
        if not v.strip():
            raise ValueError("名前は空白のみにはできません")
        return v.strip()
```

### `min_length` vs カスタムバリデータ

| 方法 | チェック内容 | 例 |
|------|-----------|-----|
| `min_length=1` | 長さが1以上か | `""` → NG, `" "` → OK |
| カスタムバリデータ | 空白のみでないか | `""` → NG, `" "` → NG |

**`min_length=1` だけだと空白文字（スペースだけ）は通る** 。両方使うのが安全。

### FastAPI のバリデーションエラーレスポンス

Pydantic のバリデーションエラーは FastAPI が自動的に `422` で返す:

```json
{
  "detail": [
    {
      "type": "value_error",
      "loc": ["body", "name"],
      "msg": "Value error, 名前は空白のみにはできません",
      "input": "   "
    }
  ]
}
```

## Practice（練習課題）

### 課題: 以下のスキーマにバリデーションを追加せよ

```python
class OrderCreate(BaseModel):
    product_name: str
    quantity: int
    email: str
```

要件:
- `product_name`: 必須、1-200文字、空白のみ不可
- `quantity`: 必須、1以上、1000以下
- `email`: 必須、`@` を含むこと

### 模範解答

<details>
<summary>クリックで表示</summary>

```python
from pydantic import BaseModel, Field, field_validator

class OrderCreate(BaseModel):
    product_name: str = Field(..., min_length=1, max_length=200)
    quantity: int = Field(..., ge=1, le=1000)
    email: str = Field(...)

    @field_validator("product_name")
    @classmethod
    def product_name_not_blank(cls, v):
        if not v.strip():
            raise ValueError("商品名は空白のみにはできません")
        return v.strip()

    @field_validator("email")
    @classmethod
    def email_must_contain_at(cls, v):
        if "@" not in v:
            raise ValueError("メールアドレスに @ が必要です")
        return v
```

</details>

## Mission（本番課題）

### 状況

QA チームから報告:

> 「POST /users で `{"name": "", "email": "test@example.com"}` を送ると、名前が空のユーザーが作成されちゃいます。スペースだけの名前 `{"name": "   "}` も通ります。これバリデーション入れてください。」

### やること

1. **バグを再現する**
   ```bash
   curl -X POST http://localhost:8000/users \
     -H "Content-Type: application/json" \
     -d '{"name": "", "email": "test@example.com"}'
   ```
   → 200 OK で空名前のユーザーが作成されることを確認

2. **GitHub Issue を作成する**
   - タイトル: `fix: POST /users で名前の空文字バリデーションを追加`

   <details>
   <summary>Issue の書き方例</summary>

   ```markdown
   ## 概要
   POST /users で名前の空文字・空白のみの入力を弾くバリデーションを追加する。

   ## 再現手順
   1. POST /users に {"name": "", "email": "test@example.com"} を送信
   2. 200 OK で空名前のユーザーが作成される
   3. POST /users に {"name": "   ", "email": "test@example.com"} を送信
   4. 200 OK で空白のみの名前のユーザーが作成される

   ## 期待する動作
   空文字・空白のみの名前は 422 Unprocessable Entity で拒否される。

   ## タスク
   - [ ] UserCreate スキーマに min_length=1 を追加
   - [ ] 空白のみを弾くカスタムバリデータを追加
   - [ ] UserUpdate スキーマにも同様のバリデーションを追加
   - [ ] テストを追加（空文字、空白のみ、正常系）

   ## 受入条件
   - [ ] 空文字で POST すると 422 が返る
   - [ ] 空白のみで POST すると 422 が返る
   - [ ] 正常な名前で POST すると成功する
   ```

   **ポイント**:
   - 再現手順を具体的に記載
   - 空文字と空白のみの両方をカバー
   </details>

3. **ブランチを作成する**
   - `fix/proj01-user-name-validation`

4. **バリデーションを追加する**
   - `UserCreate` に `min_length=1` を追加
   - 空白のみを弾くカスタムバリデータを追加
   - `UserUpdate` にも同様のバリデーションを追加

5. **テストを書く**
   - 空文字 `""` で 422 が返ること
   - 空白のみ `"   "` で 422 が返ること
   - 正常な名前で 200 が返ること

6. **PR を作成する**

   <details>
   <summary>PR の書き方例</summary>

   ```markdown

   POST /users で名前の空文字・空白のみの入力を弾くバリデーションを追加しました。

   Closes #XX

   - UserCreate スキーマに Field(min_length=1) を追加
   - 空白のみを弾くカスタムバリデータを追加

   - テストを追加（空文字、空白のみ、正常系）

   ## 原因
   Pydantic スキーマに min_length や空白チェックがなく、空の名前が通っていた。

   ## 修正内容
   - min_length=1 で空文字を弾く
   - カスタムバリデータで空白のみの入力を弾く

   ## テスト
   - [ ] 空文字 "" で 422 が返る
   - [ ] 空白のみ "   " で 422 が返る
   ```

### テストの参考

## Review（自己採点チェックリスト）

### 必須項目

- [ ] 空文字 `""` で POST すると 422 が返る
- [ ] 空白のみ `"   "` で POST すると 422 が返る
- [ ] 正常な名前で POST すると 200（または 201）が返る
- [ ] テストが追加されている（空文字、空白のみ、正常系）
- [ ] `UserCreate` と `UserUpdate` の両方にバリデーションがある
- [ ] GitHub Issue と PR が作成されている

### 加点項目

- [ ] エラーメッセージがわかりやすい（「名前は1文字以上必要です」など）
- [ ] `strip()` で前後の空白を除去している
- [ ] email のフォーマットバリデーションも追加した
- [ ] `max_length` も設定している

### よくある間違い

- `min_length=1` だけでスペースのみの入力を弾けていない
- `UserCreate` だけにバリデーションを入れて `UserUpdate` を忘れる
- エンドポイント側（ルートハンドラ）でバリデーションしてしまう（Pydantic に任せるべき）
- テストで 400 を期待する（FastAPI のバリデーションエラーは 422）
