## Learn（概念理解）

### バグ修正の基本フロー

実務でのバグ対応は、いきなりコードを直すのではなく **手順を踏む**:

```
1. 再現する     → 本当にバグか確認
2. 原因を特定する → コードを読んで根本原因を見つける
3. 修正する     → 最小限の変更で直す
4. テストを書く  → 同じバグが再発しないことを保証
5. PR を出す    → レビューしてもらう
```

### Issue の書き方（バグ報告）

実務では以下の情報を含める:

```markdown
## バグ概要
GET /users が DB にデータがない状態で 500 Internal Server Error を返す。

## 再現手順
1. DB を空にする（users テーブルのレコードを全削除）
2. GET /users を呼ぶ
3. → 500 Internal Server Error

## 期待する動作
空のリスト `[]` が 200 OK で返される。

## 実際の動作
500 Internal Server Error（IndexError: list index out of range）

## 環境
- Python 3.11
- FastAPI 0.104
- ローカル開発環境
```

### 防御的プログラミング

「入力やデータが想定外でもクラッシュしない」書き方:

```python
# ❌ 悪い例: 空リストで IndexError
users = db.query(User).all()
first = users[0]  # リストが空なら IndexError!

# ✅ 良い例: 空リストを正しく処理
users = db.query(User).all()
if users:
    first = users[0]
    print(f"First user: {first.name}")
return users  # 空なら [] が返る
```

### HTTP ステータスコードの基本

| コード | 意味 | 使い所 |
|-------|------|--------|
| 200 | OK | 正常に処理完了 |
| 201 | Created | リソース作成成功 |
| 400 | Bad Request | リクエストが不正 |
| 404 | Not Found | リソースが存在しない |
| 500 | Internal Server Error | サーバー側のバグ |

**500 はバグ** 。ユーザーの操作で 500 が出るのは常にサーバー側の問題。

## Practice（練習課題）

### 課題: 以下のコードのバグを見つけて修正せよ

```python
@app.get("/products")
def list_products(db: Session = Depends(get_db)):
    products = db.query(Product).all()
    # 最新の商品情報をログに出力
    latest = products[-1]
    print(f"Latest product: {latest.name}")
    return products
```

**Q1.** このコードにはどんなバグがあるか？
**Q2.** どのような条件で発生するか？
**Q3.** 修正コードを書け。

### 模範解答

<details>
<summary>クリックで表示</summary>

**A1.** `products` が空リストのとき、`products[-1]` で `IndexError` が発生し 500 エラーになる。

**A2.** DB に Product レコードが1件もない状態で `GET /products` を呼んだとき。

**A3.**
```python
@app.get("/products")
def list_products(db: Session = Depends(get_db)):
    products = db.query(Product).all()
    if products:
        latest = products[-1]
        print(f"Latest product: {latest.name}")
    return products
```

</details>

## Mission（本番課題）

### 状況

QA チームの Slack チャンネルに以下の報告が上がりました:

> 「GET /users を叩くと 500 になります。多分 DB にデータがないときだけ発生します。再現手順: DB 空の状態で `curl http://localhost:8000/users` → 500。」

### やること

1. **バグを再現する**
   - DB を空にする（または新しいDBを作る）
   - `GET /users` を叩いて 500 を確認
   ```bash
   curl http://localhost:8000/users
   ```
   - エラーログを確認する（ターミナルの出力）

2. **GitHub Issue を作成する**
   - タイトル: `fix: GET /users が空DBで500を返す`
   - 本文: 再現手順・期待する動作・実際の動作を記載

   <details>
   <summary>Issue の書き方例</summary>

   ```markdown
   ## 概要
   GET /users が DB にデータがない状態で 500 Internal Server Error を返す。

   ## 再現手順
   1. DB を空にする（users テーブルのレコードを全削除）
   2. `curl http://localhost:8000/users` を実行
   3. 500 Internal Server Error が返る

   ## 期待する動作
   空のリスト `[]` が 200 OK で返される。

   ## 実際の動作
   500 Internal Server Error
   ```
   IndexError: list index out of range
   ```

   ## 影響範囲
   - DB が空の状態での GET /users 呼び出し
   - 新規環境やテスト環境で発生する可能性

   ## 環境
   - Python 3.11
   - FastAPI 0.104
   - ローカル開発環境
   ```

   **ポイント**:
   - 再現手順を明確に記載
   - 期待する動作と実際の動作を対比
   - エラーメッセージを含める
   </details>

3. **原因を特定する**
   - `main.py` の `list_users` 関数を読む
   - どの行でエラーが発生しているか特定する

4. **ブランチを作成する**
   - `fix/proj01-list-users-empty-db`

5. **修正する**
   - 空リストでもクラッシュしないように修正
   - 空リストの場合は `[]` を返す

6. **テストを書く**
   - DB が空のときに `GET /users` が `200` + `[]` を返すテスト
   - DB にデータがあるときに正常に返すテスト

7. **PR を作成する**
   - Issue を参照する
   - 原因と修正内容を説明する

   <details>
   <summary>PR の書き方例</summary>

   ```markdown
   ## 概要
   GET /users が空DBで500エラーを返すバグを修正しました。

   Closes #XX

   ## 変更内容
   - list_users 関数で空リストの場合の処理を追加
   - 空DB時のテストケースを追加

   ## 原因
   `main.py` の `list_users` 関数で、users リストが空の場合に `users[0]` でアクセスしようとして IndexError が発生していた。

   ## 修正内容
   空リストの場合は先頭要素へのアクセスをスキップし、空リスト `[]` を返すように修正。

   ## テスト
   - [ ] 空DB時に GET /users が 200 + [] を返す
   - [ ] データありの場合も正常に動作する
   - [ ] 既存のテストが全て通る

   ## レビュー観点
   - [ ] 空リストのチェックが適切か
   - [ ] テストケースが十分か
   ```

   **ポイント**:
   - 原因を明確に説明
   - 修正内容をシンプルに記載
   - テスト追加を明示
   </details>

## Review（自己採点チェックリスト）

### 必須項目

- [ ] バグを再現できた（500 エラーを確認した）
- [ ] 原因を正しく特定した（`users[0]` による `IndexError`）
- [ ] 修正後、空 DB で `GET /users` が `200` + `[]` を返す
- [ ] 修正後、データがある場合の既存テストも通る
- [ ] 空 DB 用のテストを追加した
- [ ] GitHub Issue にバグの再現手順が記載されている
- [ ] PR が作成され、Issue を参照している

### 加点項目

- [ ] コミットメッセージが `fix:` プレフィックス
- [ ] PR 説明に「原因」と「修正内容」が明確に書かれている
- [ ] 不要なログ出力（`print(f"First user: ...")`）も修正した
- [ ] テストに正常系（データあり）と異常系（データなし）の両方がある

### よくある間違い

- バグを再現せずにコードだけ見て修正する（再現確認は必須）
- テストを書かずに修正だけする（回帰テストがないと再発リスク）
- 修正が過剰（例: 空の場合に404を返す → これは間違い。空リストは正常）
- Issue を作らずにいきなり PR を出す
