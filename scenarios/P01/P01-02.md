# P01-02: テストカバレッジの確認と改善

> **シナリオ**: TLから「テストカバレッジ確認して。多分ひどい」
> **想定時間**: 3h
> **スキルタグ**: P3（テスト設計・実装）, P8

---

## Learn（概念理解）

### テストカバレッジとは

コードのうち **何%がテストで実行されているか** を示す指標。

```
カバレッジ = テストで通った行数 / 全体の行数 × 100
```

### なぜカバレッジを測るのか

- **テストが不足している箇所** を可視化できる
- 「テスト書いたつもり」が本当にコードを網羅しているか確認できる
- チームで **最低カバレッジ基準** を設けることで品質を担保

### 目安

| カバレッジ | 評価 |
|-----------|------|
| ~30% | 危険。主要機能にテストがない可能性大 |
| 50-70% | 最低限。主要パスはカバーされている |
| 70-85% | 良好。エッジケースも一部カバー |
| 85%+ | 優秀。ただし100%を目指す必要はない |

### pytest-cov の使い方

```bash
# インストール
pip install pytest-cov

# カバレッジ付きでテスト実行
pytest --cov=. --cov-report=term-missing tests/

# HTML レポート生成
pytest --cov=. --cov-report=html tests/
# → htmlcov/index.html をブラウザで開く
```

出力例:

```
Name       Stmts   Miss  Cover   Missing
-----------------------------------------
main.py      120     85    29%   45-60, 72-95, 110-130
-----------------------------------------
TOTAL         120     85    29%
```

`Missing` の列で **テストされていない行番号** がわかる。

### テストの優先順位

全部にテストを書く時間がないとき、何から書くか:

1. **正常系の主要パス** — ユーザー作成、ユーザー取得など基本 CRUD
2. **バリデーションエラー** — 不正な入力で適切なエラーが返るか
3. **エッジケース** — 空リスト、存在しない ID、重複データ
4. **エラーハンドリング** — 500 エラーにならないか

## Practice（練習課題）

### 課題: カバレッジレポートを読む

以下のカバレッジレポートを見て、質問に答えてください:

```
Name           Stmts   Miss  Cover   Missing
---------------------------------------------
app/main.py       80     55    31%   20-35, 45-60, 70-80
app/models.py     25      0   100%
app/routes.py     60     40    33%   15-30, 40-55
---------------------------------------------
TOTAL             165     95    42%
```

**Q1.** 全体のカバレッジは何%か？
**Q2.** 最もテストが不足しているファイルはどれか？
**Q3.** テストを追加するとしたら、どのファイルから手をつけるべきか？理由も述べよ。

### 模範解答

<details>
<summary>クリックで表示</summary>

**A1.** 42%

**A2.** `app/main.py`（31%）。ただし `app/routes.py` も 33% で同程度に低い。

**A3.** `app/routes.py` から。理由:
- ルートハンドラはユーザーが直接触る部分（API エンドポイント）
- ここのバグはユーザーに直接影響する
- `main.py` はアプリの起動設定が多く、テストの優先度は routes より低い

</details>

## Mission（本番課題）

### 状況

TL から Slack:

> 「テストカバレッジ見てくれない？ `pytest-cov` 入れて測定して。多分ひどい数字だと思う。レポート見せて、今後の改善方針も考えてほしい。」

### やること

1. **GitHub Issue を作成する**
   - タイトル: `test: テストカバレッジの測定と改善方針策定`

   <details>
   <summary>Issue の書き方例</summary>

   ```markdown
   ## 概要
   テストカバレッジを測定し、現状を把握した上で改善方針を策定する。

   ## 背景
   現在のテストカバレッジが不明のため、品質の可視化と改善が必要。

   ## タスク
   - [ ] pytest-cov を導入
   - [ ] カバレッジを測定し、現状を分析
   - [ ] カバレッジ設定を pyproject.toml に追加
   - [ ] .gitignore に htmlcov/ を追加
   - [ ] 改善方針（優先順位と目標値）を策定

   ## 受入条件
   - [ ] pytest --cov でカバレッジが測定できる
   - [ ] カバレッジの数値と未テスト箇所が明確
   - [ ] 今後の改善方針が文書化されている

   ## 参考資料
   - pytest-cov ドキュメント
   ```

   **ポイント**:
   - カバレッジ測定の重要性を強調
   - 現状分析と改善方針の両方を含む
   </details>

2. **ブランチを作成する**
   - `test/proj01-coverage-setup`
3. **`pytest-cov` を追加する**
   - `requirements.txt` に `pytest-cov` を追加
4. **カバレッジを測定する**
   ```bash
   pytest --cov=. --cov-report=term-missing tests/
   ```
5. **結果を分析する**
   - 全体カバレッジは何%か？
   - どの行がテストされていないか？
   - 特にテストが必要な箇所はどこか？
6. **カバレッジ設定を追加する**
   - `pyproject.toml` または `setup.cfg` にカバレッジ設定を追加
   ```toml
   # pyproject.toml
   [tool.pytest.ini_options]
   addopts = "--cov=. --cov-report=term-missing"
   ```
7. **`.gitignore` に追記する**
   - `htmlcov/` ディレクトリを除外
8. **PR を作成する**
   - カバレッジの測定結果をPR説明に含める
   - 今後の改善方針を記載する

   <details>
   <summary>PR の書き方例</summary>

   ```markdown
   ## 概要
   pytest-cov を導入し、テストカバレッジの測定環境を構築しました。

   Closes #XX

   ## 変更内容
   - requirements.txt に pytest-cov を追加
   - pyproject.toml にカバレッジ設定を追加
   - .gitignore に htmlcov/ を追加

   ## 測定結果
   | File | Coverage |
   |------|----------|
   | main.py | XX% |
   | **TOTAL** | **XX%** |

   未テスト箇所:
   - GET /users の空DB時の処理
   - PUT /users/{id}
   - DELETE /users/{id}

   ## 改善方針
   1. 主要エンドポイントの正常系テストを追加（優先度: 高）
   2. エラーケースのテストを追加（優先度: 中）
   3. カバレッジ 70% を短期目標とする

   ## レビュー観点
   - [ ] カバレッジ設定が適切か
   - [ ] 改善方針が妥当か
   ```

   **ポイント**:
   - 測定結果を数値で示す
   - 未テスト箇所を具体的に列挙
   - 改善方針を優先順位付きで記載
   </details>

### 期待する PR の説明（参考）

```markdown
## Background
テストカバレッジが不明だったため、pytest-cov を導入して測定した。

## Changes
- `requirements.txt` に pytest-cov を追加
- `pyproject.toml` にカバレッジ設定を追加
- `.gitignore` に htmlcov/ を追加

## 測定結果
| File | Coverage |
|------|----------|
| main.py | XX% |
| **TOTAL** | **XX%** |

テストされていない箇所:
- GET /users（空DB時のハンドリング）
- GET /users/{id}（存在しない場合）
- PUT /users/{id}
- PATCH /users/{id}
- DELETE /users/{id}
- Teams 関連エンドポイント全般

## 今後の改善方針
1. 各エンドポイントの正常系テストを追加（優先度: 高）
2. エラーケースのテストを追加（優先度: 中）
3. カバレッジ 70% を目標とする
```

## Review（自己採点チェックリスト）

### 必須項目

- [ ] `pytest-cov` が `requirements.txt` に追加されている
- [ ] `pytest --cov` でカバレッジレポートが出力される
- [ ] カバレッジの数値を確認し、Issue または PR に記載されている
- [ ] テストされていない箇所を特定している
- [ ] GitHub Issue が作成されている
- [ ] PR が作成されている

### 加点項目

- [ ] `pyproject.toml` にカバレッジ設定が追加されている
- [ ] `.gitignore` に `htmlcov/` が追加されている
- [ ] HTML レポートを生成して確認した
- [ ] 今後の改善方針（どの順番でテストを追加するか）を記載している
- [ ] カバレッジの最低基準（例: 70%）を提案している

### よくある間違い

- `pytest-cov` をインストールしたが `requirements.txt` に追加し忘れる
- カバレッジの数値だけ見て、具体的にどこが不足しているか分析しない
- 「100% を目指す」と書く（現実的ではない。70-85% が妥当な目標）
