## Learn（概念理解）

### N+1 問題とは

**最も頻繁に発生するパフォーマンス問題**:

```python
# ❌ Bad: N+1 問題
tasks = db.query(Task).all()  # 1回のクエリ（全タスク取得）

for task in tasks:
    print(task.project.name)  # 各タスクごとに1回クエリ（N回）

# 合計: 1 + N 回のクエリ（1000タスクなら1001回！）
```

**SQL ログ**:
```sql
SELECT * FROM tasks;  -- 1回目
SELECT * FROM projects WHERE id = 1;  -- 2回目
SELECT * FROM projects WHERE id = 1;  -- 3回目（重複！）
SELECT * FROM projects WHERE id = 2;  -- 4回目
... (1000回続く)
```

### SQLAlchemy のクエリログ

```python
# ❌ Good: クエリログを有効化
import logging

logging.basicConfig()
logging.getLogger('sqlalchemy.engine').setLevel(logging.INFO)

# または環境変数
# export SQLALCHEMY_ECHO=1
```

ログ出力例:
```
INFO  SELECT tasks.id, tasks.title, tasks.project_id FROM tasks
INFO  SELECT projects.id, projects.name FROM projects WHERE projects.id = ?
INFO  SELECT projects.id, projects.name FROM projects WHERE projects.id = ?
...
```

### プロファイリングツール

```python
# 1. Python の cProfile
import cProfile
import pstats

profiler = cProfile.Profile()
profiler.enable()

# 測定対象のコード
response = client.get("/tasks")

profiler.disable()
stats = pstats.Stats(profiler)
stats.sort_stats('cumulative')
stats.print_stats(10)  # 上位10件

# 2. SQLAlchemy のイベントフック
from sqlalchemy import event
from sqlalchemy.engine import Engine
import time

query_count = 0
query_times = []

@event.listens_for(Engine, "before_cursor_execute")
def before_cursor_execute(conn, cursor, statement, parameters, context, executemany):
    global query_count
    query_count += 1
    conn.info.setdefault('query_start_time', []).append(time.time())

@event.listens_for(Engine, "after_cursor_execute")
def after_cursor_execute(conn, cursor, statement, parameters, context, executemany):
    total = time.time() - conn.info['query_start_time'].pop(-1)
    query_times.append(total)
    print(f"Query {query_count}: {total:.4f}s")
    print(f"  {statement[:100]}...")

# エンドポイント実行後
print(f"Total queries: {query_count}")
print(f"Total time: {sum(query_times):.4f}s")
```

### N+1 問題の検出方法

```
1. SQLAlchemy のクエリログを確認
2. 同じ SELECT 文が繰り返されていないか確認
3. クエリ数をカウント（期待: O(1)、実際: O(N)）
4. レスポンスタイムとクエリ数の相関を分析
```

## Practice（練習課題）

### 課題: 以下のコードの N+1 問題を検出せよ

```python
@app.get("/orders")
def list_orders(db: Session = Depends(get_db)):
    orders = db.query(Order).all()
    return [
        {
            "id": order.id,
            "user_name": order.user.name,  # ← ここで user テーブルにアクセス
            "total": order.total
        }
        for order in orders
    ]
```

**Q1.** このコードで発行される SQL クエリを列挙せよ（100注文の場合）。
**Q2.** クエリ数は何回か？
**Q3.** N+1 問題が発生しているか？

### 模範解答

<details>
<summary>クリックで表示</summary>

**A1. 発行される SQL**:
```sql
-- 1回目: 全注文を取得
SELECT * FROM orders;

-- 2回目以降: 各注文のユーザーを取得（100回）
SELECT * FROM users WHERE users.id = 1;
SELECT * FROM users WHERE users.id = 2;
SELECT * FROM users WHERE users.id = 1;  -- 重複
...
```

**A2. クエリ数**:
- 最悪: 1 + 100 = 101回（全ユーザーが異なる場合）
- 最良: 1 + ユニークユーザー数

**A3. N+1 問題**:
✅ 発生している。`order.user.name` で各注文ごとにクエリが発行される。

</details>

## Mission（本番課題）

### 状況

P05-01 でベースラインを測定した結果、GET /tasks が 3.2秒と判明しました。

TL から:

> 「次は原因を特定しよう。SQLAlchemy のクエリログを有効にして、何回クエリが発行されているか確認して。N+1 問題の可能性が高い。」

### やること

1. **SQLAlchemy のクエリログを有効化する**
   - `database.py` または `main.py` に設定追加
   - logging レベルを INFO に設定

2. **GET /tasks のクエリ数をカウントする**
   - イベントフックで自動カウント
   - または手動でログを集計

3. **クエリログを分析する**
   - 同じ SELECT 文が繰り返されていないか確認
   - どのリレーションで N+1 が発生しているか特定

4. **問題箇所を特定する**
   - コードのどの行で N+1 が発生しているか
   - `task.project` または `task.assignee` へのアクセス

5. **分析結果を文書化する**
   - ファイル: `docs/performance-analysis.md`
   - 問題の特定結果
   - クエリ数（期待 vs 実際）
   - ボトルネックの箇所

6. **PR を作成する**
   - 分析結果を含める
   - 次のステップ（P05-03 で修正）を明記
