> **想定時間**: 5h
> **スキルタグ**: P7（パフォーマンス）, P2（DB設計）

## Learn（概念理解）

### インデックスとは

**検索を高速化するデータ構造**（本の索引と同じ）:

```sql
-- インデックスなし: フルテーブルスキャン（全行を検索）
SELECT * FROM tasks WHERE status = 'TODO';
-- 10,000行全てをスキャン → 遅い（100ms）

-- インデックスあり: インデックススキャン
CREATE INDEX idx_tasks_status ON tasks(status);
SELECT * FROM tasks WHERE status = 'TODO';
-- インデックスで該当行を直接取得 → 速い（5ms）
```

### EXPLAIN ANALYZE

**クエリ実行計画を確認する**ツール:

```sql
EXPLAIN ANALYZE SELECT * FROM tasks WHERE status = 'TODO';

-- インデックスなし
Seq Scan on tasks  (cost=0.00..180.00 rows=3333 width=100) (actual time=0.012..2.345 rows=3333 loops=1)
  Filter: (status = 'TODO'::task_status)
  Rows Removed by Filter: 6667
Planning Time: 0.123 ms
Execution Time: 2.456 ms

-- インデックスあり
Index Scan using idx_tasks_status on tasks  (cost=0.29..120.00 rows=3333 width=100) (actual time=0.015..0.234 rows=3333 loops=1)
  Index Cond: (status = 'TODO'::task_status)
Planning Time: 0.098 ms
Execution Time: 0.289 ms
```

**見方**:
- **Seq Scan**: フルテーブルスキャン（遅い）
- **Index Scan**: インデックススキャン（速い）
- **Execution Time**: 実際の実行時間

### インデックスを追加すべき列

```
✅ インデックスが有効な列:
- WHERE 句で頻繁に使用される列（status, priority, project_id）
- JOIN で使用される列（外部キー）
- ORDER BY で使用される列（created_at）

❌ インデックスが不要な列:
- カーディナリティが低い列（性別: 2値、真偽値）
- 頻繁に更新される列（インデックス更新のオーバーヘッド）
- テーブルサイズが小さい（<1000行）
```

### 複合インデックス

```sql
-- 単一列インデックス
CREATE INDEX idx_tasks_status ON tasks(status);

-- 複合インデックス（複数列）
CREATE INDEX idx_tasks_status_priority ON tasks(status, priority);

-- 使用例
SELECT * FROM tasks WHERE status = 'TODO' AND priority = 'HIGH';
-- idx_tasks_status_priority を使用（最速）

SELECT * FROM tasks WHERE status = 'TODO';
-- idx_tasks_status_priority も使用可能（先頭列だけでOK）

SELECT * FROM tasks WHERE priority = 'HIGH';
-- idx_tasks_status_priority は使用不可（先頭列がない）
```

### Alembic でインデックス追加

```python
# alembic/versions/xxxx_add_indexes.py
def upgrade():
    # 単一列インデックス
    op.create_index('idx_tasks_status', 'tasks', ['status'])
    op.create_index('idx_tasks_priority', 'tasks', ['priority'])
    op.create_index('idx_tasks_project_id', 'tasks', ['project_id'])

    # 複合インデックス
    op.create_index('idx_tasks_status_priority', 'tasks', ['status', 'priority'])

def downgrade():
    op.drop_index('idx_tasks_status', table_name='tasks')
    op.drop_index('idx_tasks_priority', table_name='tasks')
    op.drop_index('idx_tasks_project_id', table_name='tasks')
    op.drop_index('idx_tasks_status_priority', table_name='tasks')
```

## Practice（練習課題）

### 課題: 以下のクエリにインデックスを追加せよ

```sql
-- よく実行されるクエリ
SELECT * FROM products WHERE category = 'Electronics' AND price > 100 ORDER BY created_at DESC;
```

**Q1.** どの列にインデックスを追加すべきか？
**Q2.** 複合インデックスは必要か？
**Q3.** マイグレーションを書け。

### 模範解答

<details>
<summary>クリックで表示</summary>

**A1. インデックスを追加すべき列**:
- `category`: WHERE句で使用
- `price`: WHERE句で使用
- `created_at`: ORDER BY で使用

**A2. 複合インデックス**:
推奨: `(category, price, created_at)` の複合インデックス
理由: WHERE と ORDER BY の両方をカバー

**A3. マイグレーション**:

```python
def upgrade():
    # 複合インデックス（最も効果的）
    op.create_index(
        'idx_products_category_price_created',
        'products',
        ['category', 'price', 'created_at']
    )

def downgrade():
    op.drop_index('idx_products_category_price_created', table_name='products')
```

</details>

## Mission（本番課題）

### 状況

N+1とSQL集計を改善した後、TL から:

> 「EXPLAIN ANALYZE でクエリ実行計画を確認してみたら、tasks テーブルでフルスキャンが発生してる。インデックスを追加しよう。」

### やること

1. **EXPLAIN ANALYZE で確認する**
   ```sql
   EXPLAIN ANALYZE SELECT * FROM tasks WHERE status = 'TODO';
   EXPLAIN ANALYZE SELECT * FROM tasks WHERE project_id = 1 AND status = 'TODO';
   ```

2. **フルスキャンを特定する**
   - Seq Scan が発生している箇所
   - 実行時間が長いクエリ

3. **インデックスを設計する**
   - WHERE句で使用される列を特定:
     - `status`
     - `priority`
     - `project_id`（外部キー）
     - `assignee_id`（外部キー）
   - 複合インデックスの必要性を検討

4. **Alembic マイグレーションを作成する**
   - ファイル: `alembic/versions/xxxx_add_task_indexes.py`
   - インデックス:
     - `idx_tasks_status`
     - `idx_tasks_priority`
     - `idx_tasks_project_id`
     - `idx_tasks_assignee_id`
     - `idx_tasks_status_priority`（複合）

5. **マイグレーションを適用する**
   ```bash
   alembic revision -m "add indexes to tasks table"
   alembic upgrade head
   ```

6. **EXPLAIN ANALYZE で効果を確認する**
   - Seq Scan → Index Scan に変わったか
   - Execution Time の改善

7. **パフォーマンスを測定する**
   - フィルタリングクエリの速度
   - Before/After比較

8. **performance-baseline.md を更新する**

9. **PR を作成する**

## Review（自己採点チェックリスト）

### 必須項目

- [ ] EXPLAIN ANALYZE でフルスキャンを確認した
- [ ] 適切な列にインデックスを追加した（status, priority, 外部キー）
- [ ] Alembic マイグレーションが作成・適用されている
- [ ] EXPLAIN ANALYZE で Index Scan を確認した
- [ ] パフォーマンスが改善されている（50ms → 5ms）
- [ ] performance-baseline.md が更新されている

### 加点項目

- [ ] 複合インデックスを追加している
- [ ] Before/After の EXPLAIN ANALYZE 結果を文書化している
- [ ] インデックスのサイズも考慮している
- [ ] 不要なインデックスを避けている

### よくある間違い

- EXPLAIN ANALYZE を実行しない（推測だけでインデックス追加）
- 全ての列にインデックスを追加する（オーバーヘッド増加）
- 複合インデックスの列順を間違える（効果が出ない）
- downgrade を実装しない（ロールバック不可）
- パフォーマンスを測定しない（効果未確認）
