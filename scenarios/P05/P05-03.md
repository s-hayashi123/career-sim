## Learn（概念理解）

### Eager Loading（即時読み込み）

N+1 問題の解決策は **Eager Loading**:

```python
# ❌ Bad: Lazy Loading（遅延読み込み）→ N+1
tasks = db.query(Task).all()
for task in tasks:
    print(task.project.name)  # 各タスクごとにクエリ

# ✅ Good: Eager Loading（即時読み込み）
from sqlalchemy.orm import joinedload

tasks = db.query(Task).options(joinedload(Task.project)).all()
for task in tasks:
    print(task.project.name)  # クエリ済みのデータを使用

# 発行されるクエリ: 1回のJOIN
# SELECT tasks.*, projects.* FROM tasks LEFT OUTER JOIN projects ON tasks.project_id = projects.id
```

### joinedload vs selectinload

| 方式 | JOIN方法 | クエリ数 | 用途 |
|------|----------|---------|------|
| **joinedload** | LEFT OUTER JOIN | 1回 | 1対多で「多」が少ない場合 |
| **selectinload** | WHERE IN | 2回 | 1対多で「多」が多い場合 |

```python
# joinedload: 1回のJOIN
tasks = db.query(Task).options(joinedload(Task.project)).all()
# SELECT tasks.*, projects.* FROM tasks LEFT OUTER JOIN projects ...

# selectinload: 2回のクエリ
tasks = db.query(Task).options(selectinload(Task.project)).all()
# 1回目: SELECT * FROM tasks
# 2回目: SELECT * FROM projects WHERE id IN (1, 2, 3, ...)
```

**選び方**:
- **joinedload**: 1対1、または1対多で「多」が少ない
- **selectinload**: 1対多で「多」が多い（重複行を避ける）

### 複数のリレーションを読み込む

```python
# 複数の options を指定
tasks = (
    db.query(Task)
    .options(joinedload(Task.project))  # project を読み込む
    .options(joinedload(Task.assignee))  # assignee を読み込む
    .all()
)

# または
from sqlalchemy.orm import joinedload

tasks = (
    db.query(Task)
    .options(
        joinedload(Task.project),
        joinedload(Task.assignee)
    )
    .all()
)
```

### サービス層での適用

```python
# services/task_service.py
from sqlalchemy.orm import joinedload

class TaskService:
    def list_tasks(self, filters: TaskFilters) -> list[Task]:
        query = self.db.query(Task)

        # Eager Loading
        query = query.options(
            joinedload(Task.project),
            joinedload(Task.assignee)
        )

        # フィルタリング
        if filters.status:
            query = query.filter(Task.status == filters.status)

        # ソート
        query = query.order_by(Task.created_at.desc())

        return query.all()
```

### ADR-008 での記録

```markdown
# ADR-008: N+1問題の解消戦略

## Context
GET /tasks で N+1 問題が発生（1001クエリ）。
task.project と task.assignee へのアクセスで毎回クエリ。

## Decision
Eager Loading を採用:
- joinedload を使用（1回の JOIN で取得）
- サービス層の list_tasks で適用

## Alternatives
1. selectinload: 2回のクエリ（今回は不要）
2. subqueryload: サブクエリ（複雑）

## Consequences
**Pros**:
- クエリ数: 1001回 → 1回（99.9%削減）
- レスポンスタイム: 3.2s → 0.3s（90%改善）

**Cons**:
- JOIN のオーバーヘッド（軽微）
```

        }
        for order in orders

**Q1.** joinedload を使ってクエリを最適化せよ。
**Q2.** 最適化前後のクエリ数を比較せよ（100注文の場合）。

from sqlalchemy.orm import joinedload

class TaskService:
    def list_tasks(
        self,
        status: TaskStatus | None = None,
        limit: int = 100,
        offset: int = 0
    ) -> dict:
        query = self.db.query(Task)

        # Eager Loading（N+1 解消）
        query = query.options(
            joinedload(Task.project),
            joinedload(Task.assignee)
        )

        # フィルタリング
        if status:
            query = query.filter(Task.status == status)

        # ソート
        query = query.order_by(Task.created_at.desc())

        # 総件数
        total = query.count()

        # ページネーション
        tasks = query.limit(limit).offset(offset).all()

        return {
            "items": tasks,
            "total": total,
            "limit": limit,
            "offset": offset
        }

```markdown
# ADR-008: N+1問題の解消戦略

## Status
Accepted

## Context

### 問題
GET /tasks で N+1 問題が発生:
- タスク数: 1000
- クエリ数: 1001回（1 + 1000）
- レスポンスタイム: 3.2s

### 原因
Pydantic の TaskResponse が project: ProjectResponse を含むため、
各タスクごとに task.project へのアクセスが発生し、毎回クエリが実行される。

## Decision

### Eager Loading の採用
SQLAlchemy の `joinedload` を使用してリレーションを即時読み込み。

```python
query = query.options(
    joinedload(Task.project),
    joinedload(Task.assignee)
)
```

### 発行されるクエリ
```sql
SELECT tasks.*, projects.*, users.*
FROM tasks
LEFT OUTER JOIN projects ON tasks.project_id = projects.id
LEFT OUTER JOIN users ON tasks.assignee_id = users.id
```

## Alternatives

### selectinload
- 2回のクエリ（SELECT ... WHERE IN）
- 今回は joinedload で十分

### subqueryload
- サブクエリを使用
- 複雑で不要

## Consequences

### Pros
- **クエリ数**: 1001回 → 1回（99.9%削減）
- **レスポンスタイム**: 3.2s → 0.3s（90%改善）
- コードの変更が最小限（サービス層のみ）

### Cons
- JOIN のオーバーヘッド（軽微、<10ms）
- メモリ使用量が若干増加（許容範囲）

### リスク
- なし（既存のテストが全て通る）
```
