## Learn（概念理解）

### アプリケーション層 vs DB層 ページネーション

```python
# ❌ Bad: アプリケーション層（全データ取得してスライス）
tasks = db.query(Task).all()  # 10,000件取得
paginated = tasks[offset:offset+limit]  # Python でスライス
# 問題: 10,000件全てをメモリに読み込む

# ✅ Good: DB層（LIMIT/OFFSET）
tasks = db.query(Task).limit(limit).offset(offset).all()
# DB側で100件だけ取得
```

### LIMIT / OFFSET の最適化

```sql
-- 基本的なページネーション
SELECT * FROM tasks ORDER BY created_at DESC LIMIT 100 OFFSET 0;    -- 1ページ目
SELECT * FROM tasks ORDER BY created_at DESC LIMIT 100 OFFSET 100;  -- 2ページ目

-- 問題: OFFSET が大きいと遅い
SELECT * FROM tasks ORDER BY created_at DESC LIMIT 100 OFFSET 10000;  -- 10,001件スキャン
```

### カーソルベースページネーション（Keyset Pagination）

OFFSETを避ける手法:

```sql
-- 初回
SELECT * FROM tasks WHERE created_at < '2024-01-01' ORDER BY created_at DESC LIMIT 100;

-- 次ページ（最後の created_at を使用）
SELECT * FROM tasks WHERE created_at < '2023-12-31 10:00:00' ORDER BY created_at DESC LIMIT 100;
```

**メリット**:
- OFFSET不要（常に高速）
- 大量ページでもパフォーマンス一定

**デメリット**:
- ページ番号でのジャンプ不可
- 実装が複雑

### ADR-009 での戦略記録

```markdown
# ADR-009: ページネーション戦略

## Context
- 現状: Python側でスライス（全データ取得）
- 問題: 10,000タスク全てをメモリに読み込む

## Decision
LIMIT/OFFSET 方式を採用

### 実装
```sql
SELECT * FROM tasks ORDER BY created_at DESC LIMIT {limit} OFFSET {offset}
```

## Alternatives
1. カーソルベース（Keyset）: 実装複雑、ページ番号不可
2. Python スライス: パフォーマンス問題

## Consequences
**Pros**:
- メモリ使用量削減（10,000件 → 100件）
- シンプルな実装
- ページ番号でのジャンプ可能

**Cons**:
- OFFSET が大きいと遅い（10,000ページ目など）
- データ追加/削除でページずれの可能性

**将来の改善**:
- OFFSET > 1000 でカーソルベースに切り替え
```

### 模範解答

<details>
<summary>クリックで表示</summary>

```python
from fastapi import Query

@app.get("/products")
def list_products(
    limit: int = Query(default=100, le=100),
    offset: int = Query(default=0, ge=0),
    db: Session = Depends(get_db)
):
    # 総件数
    total = db.query(func.count(Product.id)).scalar()

    # ページネーション（DB側）
    products = (
        db.query(Product)
        .order_by(Product.created_at.desc())
        .limit(limit)
        .offset(offset)
        .all()
    )

    return {
        "items": products,
        "total": total,
        "limit": limit,
        "offset": offset,
        "has_next": offset + limit < total
    }
```

```python
# services/task_service.py
class TaskService:
    def list_tasks(
        self,
        status: TaskStatus | None = None,
        priority: TaskPriority | None = None,
        limit: int = 100,
        offset: int = 0
    ) -> dict:
        query = self.db.query(Task)

        # Eager Loading
        query = query.options(
            joinedload(Task.project),
            joinedload(Task.assignee)
        )

        # フィルタリング
        if status:
            query = query.filter(Task.status == status)
        if priority:
            query = query.filter(Task.priority == priority)

        # ソート
        query = query.order_by(Task.created_at.desc())

        # 総件数（フィルタ適用後）
        total = query.count()

        # ページネーション（DB側）
        tasks = query.limit(limit).offset(offset).all()

        return {
            "items": tasks,
            "total": total,
            "limit": limit,
            "offset": offset,
            "has_next": offset + limit < total
        }
```

<summary>ADR-009 の例</summary>

```markdown
# ADR-009: ページネーション戦略

## Status
Accepted

## Context

### 現状の問題
```python
tasks = db.query(Task).all()  # 10,000件全て取得
paginated = tasks[offset:offset+limit]  # Python でスライス
```

- 全データ（10,000タスク）をメモリに読み込む
- メモリ使用量: 数MB
- レスポンスタイム: 遅い（全データ転送）

## Decision

### LIMIT/OFFSET 方式を採用
```python
tasks = db.query(Task).limit(limit).offset(offset).all()
```

### 実装詳細
- DB側で LIMIT/OFFSET を使用
- 総件数は別途 COUNT クエリで取得
- デフォルト limit=100, offset=0

## Alternatives

### 1. カーソルベースページネーション
```python
query.filter(Task.created_at < last_seen_at).limit(limit)
```

**Pros**:
- OFFSET が大きくても高速
- データ追加/削除の影響を受けにくい

**Cons**:
- ページ番号でのジャンプ不可
- 実装が複雑
- 現状の要件には過剰

### 2. Python スライス（現状）
**Cons**:
- 全データをメモリに読み込む（パフォーマンス問題）

## Consequences

### Pros
- **メモリ使用量削減**: 10,000件 → 100件（99%削減）
- **シンプル**: 実装が容易
- **ページ番号対応**: UI でページ番号表示可能

### Cons
- **大きなOFFSETで遅い**: OFFSET=10000 → 10,001件スキャン
- **データずれ**: データ追加/削除でページ内容が変わる可能性

### 将来の改善案
- OFFSET > 1000 の場合、カーソルベースに切り替え
- インデックス最適化（created_at）
```
