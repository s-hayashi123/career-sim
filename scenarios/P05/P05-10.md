## Learn（概念理解）

### 5回目のレトロで意識すること

Project 01〜04 との比較:
- **P01**: 環境構築・基本ワークフロー
- **P02**: バグ修正・再発防止
- **P03**: 機能開発・設計
- **P04**: リファクタリング・コード品質
- **P05**: パフォーマンス改善 ← 今回

**成長の軌跡**:
- 100本ノックの半分（5/10プロジェクト）を完了
- 基礎 → 応用 → 最適化のサイクルを体験
- スキルの幅と深さの拡大

### パフォーマンス改善の特徴

他のプロジェクトとの違い:

| 側面 | P01-P04 | P05（パフォーマンス） |
|-----|--------|---------------------|
| 目標 | 機能実装、品質向上 | 速度・効率の向上 |
| 測定 | テスト通過、バグ数 | レスポンスタイム、クエリ数 |
| アプローチ | コード追加・修正 | 測定 → 分析 → 最適化 |
| 検証 | 動作確認 | Before/After 比較 |
| ドキュメント | ADR、README | ADR + パフォーマンスレポート |

**パフォーマンス改善で重要なこと**:
1. **定量化**: 感覚ではなく数値で判断
2. **ボトルネックの特定**: 推測ではなく計測
3. **優先順位**: 高インパクト × 低コストから
4. **トレードオフ**: 速度 vs メモリ vs 複雑さ
5. **継続的な監視**: 改善は一度きりではない

### 振り返りの深さ

5回目のレトロでは、より深い振り返りを:

```
P01-P02: 基本的な気づき（「テストが大事」「ドキュメントは必要」）
P03-P04: パターンの発見（「設計が重要」「コード品質が開発速度に影響」）
P05: 体系的な理解（「測定 → 分析 → 改善のサイクル」「データドリブンな意思決定」）
```

**Try の進化**:
- P01: 「次はテストを書く」
- P03: 「設計を先に考える」
- P05: 「パフォーマンステストを CI に組み込む」← より具体的・システマティック

P05 のスコアカードには以下を追加:

|-----|---------|

## Good（うまくいったこと）

1. **測定駆動のアプローチ**
   - ベースライン測定 → 分析 → 改善 → 検証のサイクルが確立できた
   - 推測ではなくデータに基づいた意思決定ができた

   - タスク一覧: 3.2s → 0.15s（95%改善）
   - サマリーAPI: 1.5s → 0.01s（99%改善）
   - 全エンドポイントが目標値（<200ms）を達成

3. **N+1問題の理解と解決**
   - クエリログでの検出方法を習得
   - joinedload/selectinload の使い分けができるようになった
   - 101回 → 1回のクエリ削減を実現

   - Eager Loading 戦略（ADR-008）
   - ページネーション戦略（ADR-009）

   - performance-report.md で全成果を可視化

## Bad（うまくいかなかったこと）

1. **プロファイリングツールの習熟不足**
   - 手動でのクエリカウントに頼った
   - 本格的な APM ツール（New Relic など）の経験が不足
   - より効率的な分析手法があったかもしれない

2. **負荷テストの未実施**
   - 同時リクエストでの性能を測定していない
   - スループット（req/s）が不明
   - 本番環境での性能予測が不十分

3. **キャッシュ戦略の粗さ**
   - 全キャッシュクリアで無関係なプロジェクトにも影響
   - プロジェクトIDごとの細かい無効化ができていない
   - Redis などの外部キャッシュの経験がない

4. **継続的な監視の仕組み不足**
   - 改善後の性能劣化を検知する仕組みがない
   - スロークエリログの監視が未実装
   - アラート設定がない
