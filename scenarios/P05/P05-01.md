## Learn（概念理解）

### パフォーマンス改善の基本フロー

```
1. 測定 → 現状を数値で把握（ベースライン）
2. 分析 → ボトルネックを特定
3. 改善 → 最適化を実装
4. 検証 → 改善効果を測定
5. 記録 → Before/After を文書化
```

**測定なしに改善なし**。まず現状を測定することが最重要。

### レスポンスタイムの測定

```python
import time
from fastapi import Request

# ❌ Bad: 手動で測定
start = time.time()
# ... 処理 ...
end = time.time()
print(f"Time: {end - start}s")

# ✅ Good: ミドルウェアで自動測定
@app.middleware("http")
async def add_process_time_header(request: Request, call_next):
    start_time = time.time()
    response = await call_next(request)
    process_time = time.time() - start_time
    response.headers["X-Process-Time"] = str(process_time)
    print(f"{request.method} {request.url.path} - {process_time:.3f}s")
    return response
```

### テストデータの生成（シードスクリプト）

大量データでの性能テストには**シードスクリプト**が必要:

```python
# scripts/seed.py
from models import User, Project, Task
from database import SessionLocal

def seed_data(num_projects=100, num_tasks_per_project=100):
    """テストデータを生成"""
    db = SessionLocal()

    print(f"Creating {num_projects} projects...")
    for i in range(num_projects):
        project = Project(
            name=f"Project {i}",
            description=f"Description {i}"
        )
        db.add(project)
        db.flush()  # ID を取得

        # 各プロジェクトにタスクを追加
        for j in range(num_tasks_per_project):
            task = Task(
                project_id=project.id,
                title=f"Task {j} in Project {i}",
                status="TODO" if j % 3 == 0 else ("IN_PROGRESS" if j % 3 == 1 else "DONE"),
                priority="HIGH" if j % 3 == 0 else ("MEDIUM" if j % 3 == 1 else "LOW")
            )
            db.add(task)

        if i % 10 == 0:
            db.commit()
            print(f"  {i} projects created...")

    db.commit()
    print(f"✅ Created {num_projects} projects and {num_projects * num_tasks_per_project} tasks")
    db.close()

if __name__ == "__main__":
    seed_data(100, 100)  # 10,000 タスク
```

## Mission（本番課題）

### 状況

PM から以下の報告が来ました:

> 「タスク一覧（GET /tasks）が遅すぎます。ユーザーから「3秒以上待たされる」とクレームが来ています。タスクが1000件超えてから顕著になりました。原因を調査して改善してください。」

TL からの指示:

   - 関数: `seed_data(num_projects=100, num_tasks_per_project=100)`
   - 合計 10,000 タスクを生成
   - 進捗表示（10 projects ごと）

4. **シードを実行する**
   ```bash
   python scripts/seed.py
   ```

5. **タイミングミドルウェアを追加する**
   - `main.py` に追加
   - レスポンスヘッダー `X-Process-Time` に処理時間を記録

<summary>ベースライン測定の方法</summary>

```bash
# 手動測定
curl -w "\nTime: %{time_total}s\n" http://localhost:8000/tasks

# Python スクリプトで測定
import requests
import time

def measure(url, n=5):
    times = []
    for i in range(n):
        start = time.time()
        response = requests.get(url)
        end = time.time()
        times.append(end - start)
        print(f"  {i+1}: {end - start:.3f}s")
    avg = sum(times) / len(times)
    print(f"Average: {avg:.3f}s")

measure("http://localhost:8000/tasks", 5)
```

## Review（自己採点チェックリスト）

### 必須項目

### 加点項目

- [ ] 複数エンドポイントを測定している
- [ ] 平均・最大・最小値を記録している
- [ ] 問題点が明確に特定されている（どのエンドポイントが遅いか）
- [ ] 次のステップが明記されている

### よくある間違い

- シードスクリプトで commit を忘れる（データが保存されない）
- 測定を1回だけしか行わない（ばらつきを考慮していない）
- ベースラインを記録しない（改善効果が測定できない）
- タイミングミドルウェアを本番環境に残す（オーバーヘッド）
- 環境情報を記録しない（再現性がない）
