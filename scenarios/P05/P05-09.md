## Learn（概念理解）

### パフォーマンスレポートとは

**改善活動の成果を可視化する**文書:

```
目的:
1. Before/After の定量的な比較
2. 改善施策の効果測定
3. 残存する課題の明確化
4. 次のアクションプランの策定
5. ステークホルダーへの報告
```

### 良いレポートの条件

```markdown
✅ 良いレポート:
- 定量的なデータ（%、ms、倍率）
- Before/After が明確
- 各施策の効果が個別に示されている
- 累積効果も示されている
- 残存課題と次のステップが明記されている

❌ 悪いレポート:
- 「速くなった」だけ（数値なし）
- Before が不明確
- 複数の改善を混ぜて効果不明
- 次のアクションがない
```

### パフォーマンス改善の記録フォーマット

```markdown
## エンドポイント: GET /tasks

### Before
- レスポンスタイム: 3.2s
- クエリ数: 101回（N+1問題）
- メモリ使用量: 150MB

### 施策
1. Eager Loading 導入（P05-03）
2. SQL集計化（P05-04）
3. インデックス追加（P05-05）

### After
- レスポンスタイム: 0.15s
- クエリ数: 1回
- メモリ使用量: 5MB

### 改善率
- **レスポンスタイム**: 95%改善（3.2s → 0.15s）
- **クエリ数**: 99%削減（101回 → 1回）
- **メモリ使用量**: 97%削減（150MB → 5MB）
```

### メトリクスの種類

| メトリクス | 測定方法 | 目標値 |
|-----------|---------|-------|
| レスポンスタイム | ミドルウェア測定 | <200ms |
| クエリ数 | SQLAlchemy ログ | 最小化 |
| メモリ使用量 | Python profiler | 最小化 |
| スループット | 負荷テスト | >100 req/s |
| データベースサイズ | `SELECT pg_size_pretty(pg_database_size('atlas'));` | 監視 |

### 改善施策の優先順位

```
1. 高インパクト × 低コスト: N+1解消、インデックス追加
2. 高インパクト × 高コスト: キャッシュ導入、パーティショニング
3. 低インパクト × 低コスト: コード最適化
4. 低インパクト × 高コスト: 避ける
```

## Practice（練習課題）

### 課題: 以下の改善データからレポートを作成せよ

```
施策1: インデックス追加
- Before: 500ms
- After: 50ms

施策2: キャッシュ導入
- Before: 50ms（施策1の後）
- After: 5ms

施策3: ページネーション
- Before: メモリ100MB
- After: メモリ10MB
```

**Q1.** 各施策の改善率を計算せよ。
**Q2.** 累積改善率（施策1開始前 → 施策3完了後）を計算せよ。
**Q3.** 表形式で整理せよ。

<details>
<summary>performance-report.md の構造例</summary>

```markdown
# パフォーマンス改善レポート - Project 05

## エグゼクティブサマリー

Project 05 では、タスク管理システムのパフォーマンスボトルネックを特定し、8つの改善施策を実施しました。

**主な成果**:
- **タスク一覧API**: 3.2s → 0.5s（**84%改善**）
- **サマリーAPI**: 1.5s → 0.01s（**99%改善**）
- **クエリ数削減**: 101回 → 1回（**99%削減**）
- **メモリ使用量**: 10MB → 0.5MB（**95%削減**）

全てのエンドポイントが目標値（<200ms）を達成しました。

---

## 測定環境

- データ件数: 1,000タスク、100プロジェクト、10ユーザー
- 測定方法: タイミングミドルウェア（X-Process-Time ヘッダー）
- DB: PostgreSQL 15
- Python: 3.11

---

## 改善施策の詳細

### P05-01: パフォーマンスベースライン測定

**目的**: 現状の性能を定量的に把握

**実施内容**:
- シードスクリプトでテストデータ生成（1,000タスク）
- タイミングミドルウェア実装
- 各エンドポイントのベースライン測定

**結果**:
| エンドポイント | レスポンスタイム | クエリ数 |
|--------------|----------------|---------|
| GET /tasks | 3.2s | 101回 |
| GET /projects/{id}/summary | 1.5s | 1回 |

---

### P05-02: プロファイリング（N+1問題の特定）

**目的**: ボトルネックの原因を特定

**実施内容**:
- SQLAlchemy クエリロギング有効化
- クエリカウンター実装
- N+1問題を発見（project, assignee の遅延読み込み）

**結果**:
- タスク一覧で 101回のクエリ（1回メイン + 100回 project + 100回 assignee）

---

### P05-03: N+1問題の解消

**目的**: 無駄なクエリを削減

**実施内容**:
- `joinedload(Task.project)` でプロジェクトを一括取得
- `joinedload(Task.assignee)` で担当者を一括取得
- ADR-008: Eager Loading 戦略

**Before**:
- クエリ数: 101回
- レスポンスタイム: 3.2s

**After**:
- クエリ数: 1回
- レスポンスタイム: 0.5s

**改善率**: **84%改善**（3.2s → 0.5s）、**99%削減**（101回 → 1回）

---

### P05-04: SQL集計化

**目的**: Python集計をDB側で実行

**実施内容**:
- サマリーAPIの集計を Python ループ → SQL GROUP BY に変更
- `func.count()` で集計
- メモリ使用量削減（全タスク読み込み不要）

**Before**:
- 全タスク取得: 10,000行
- レスポンスタイム: 1.5s
- メモリ: 数MB

**After**:
- 集計結果のみ: 3行
- レスポンスタイム: 0.05s
- メモリ: 数KB

**改善率**: **96%改善**（1.5s → 0.05s）

---

### P05-05: インデックス追加

**目的**: フルテーブルスキャンを解消

**実施内容**:
- EXPLAIN ANALYZE でフルスキャンを確認
- インデックス追加:
  - `idx_tasks_status`
  - `idx_tasks_priority`
  - `idx_tasks_project_id`
  - `idx_tasks_assignee_id`
  - `idx_tasks_status_priority`（複合）

**Before**:
- Seq Scan（フルスキャン）
- フィルタリング: 50ms

**After**:
- Index Scan
- フィルタリング: 5ms

**改善率**: **90%改善**（50ms → 5ms）

---

### P05-06: DB側ページネーション

**目的**: メモリ使用量削減

**実施内容**:
- Python スライス → DB の LIMIT/OFFSET
- 全データ取得を回避
- ADR-009: ページネーション戦略

**Before**:
- 全タスク読み込み: 10,000行
- メモリ: 10MB

**After**:
- ページ分のみ読み込み: 100行
- メモリ: 0.5MB

**改善率**: **95%削減**（10MB → 0.5MB）

---

### P05-07: データ保持ポリシー

**目的**: ストレージの無限増加を防ぐ

**実施内容**:
- 90日保持ポリシー策定
- 定期削除スクリプト作成
- cron 設定（毎日 2:00 AM）
- ADR-010: データ保持ポリシー

**結果**:
- ディスク使用量の増加を抑制
- 古いログの定期削除（自動化）

---

### P05-08: キャッシュ導入

**目的**: 読み取り専用APIの高速化

**実施内容**:
- `lru_cache` でサマリー計算をキャッシュ
- タスク更新時にキャッシュ無効化
- DB セッションをキャッシュキーから除外

**Before**（SQL集計後）:
- レスポンスタイム: 0.05s（毎回計算）

**After**:
- 1回目: 0.05s（計算 + キャッシュ保存）
- 2回目以降: 0.01s（キャッシュヒット）

**改善率**: **80%改善**（0.05s → 0.01s、2回目以降）

## 残存課題

### 短期的な課題

1. **大きなOFFSETのパフォーマンス**
   - 現状: LIMIT/OFFSET 方式
   - 問題: OFFSET=10000 で遅い
   - 対策: カーソルベースページネーションへの移行（OFFSET > 1000 の場合）

2. **キャッシュ戦略の洗練**
   - 現状: 全キャッシュクリア
   - 問題: 無関係なプロジェクトのキャッシュも削除
   - 対策: プロジェクトIDごとのキャッシュ無効化

### 長期的な課題

1. **Redis への移行**
   - インメモリキャッシュ → Redis
   - 複数サーバー間でのキャッシュ共有
   - TTL の柔軟な設定

2. **読み取りレプリカの導入**
   - 読み取り専用クエリをレプリカに分散
   - マスターの負荷軽減

3. **データベースパーティショニング**
   - 大規模データ（100万行以上）での性能維持
   - 月ごとにパーティション分割
```
